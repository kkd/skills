# アジャイル/XPと構造保存変容の統合

## TDD × 構造保存変容

### Red-Green-Refactor を「変容の約束」として読み直す

| TDDのフェーズ | 構造保存変容での意味 |
|-------------|-------------------|
| **Red**（テストを書く） | 変容前のセンターへの「約束」を記述する |
| **Green**（最小実装で通す） | 最小変容で約束を果たす（スモールステップ） |
| **Refactor**（整理する） | 全体性を強化する方向にのみ整理する |

**Refactorフェーズの制約**:
リファクタリングは「コードをきれいにする」ことではなく、
「センターの生命の質を高める」方向にのみ行う。

問うべき問い: 「この整理は、どの15特性を強化しているか？」

## スプリント計画への統合

### バックログアイテムの拡張フォーマット

```
## ストーリー: [タイトル]

ユーザー価値: As a [ユーザー], I want to [機能], so that [価値]

影響するセンター:
- 主要: [センター名]（変容の方向：[特性番号]を強化）
- 副次: [センター名]（影響の予測）

構造保存の観点:
- スモールステップで分解可能か: Yes/No
- 可逆性: [ある/要注意/なし]
- アーキテクチャ境界への影響: [なし/要確認]

完了の定義（DoD）の追加条件:
- [ ] テスト通過
- [ ] 影響センターの生命の質が維持/向上している
- [ ] 全体性スナップショットを記録した
```

### スプリントレトロスペクティブへの統合

**振り返りに追加する問い**:
1. 「このスプリントで、コードベースの全体性は強まったか？」
2. 「変容を元に戻したケースはあったか？何を学んだか？」
3. 「次のスプリントで着手すべきセンターはどこか？」

## リファクタリングカタログ × 15特性

主なリファクタリング手法と、それが強化する15特性の対応：

| リファクタリング手法 | 主に強化する特性 |
|-------------------|---------------|
| Extract Method | 特性1（スケールの段階性）、特性14（シンプルさ） |
| Extract Class | 特性2（力強いセンター）、特性3（境界） |
| Move Method | 特性2（力強いセンター）の再配置 |
| Introduce Parameter Object | 特性6（よい形）、特性2（力強いセンター） |
| Replace Conditional with Polymorphism | 特性9（コントラスト）、特性4（交互反復） |
| Rename Variable/Method | 特性12（共鳴）、特性14（シンプルさ） |
| Remove Dead Code | 特性15（不可分性） |
| Introduce Null Object | 特性13（空）、特性9（コントラスト） |
| Extract Interface | 特性3（境界）、特性7（局所的シンメトリー） |
| Inline Class | 特性14（シンプルさ）、特性11（粗っぽさ）の回復 |

## コードレビューへの統合

### レビューの観点を追加する

既存の「機能の正しさ」「テストカバレッジ」に加えて：

**全体性の観点でのレビュー問い**:
1. 「このPRは、既存のセンターを強化しているか、希薄化しているか？」
2. 「変容はスモールステップで実施されているか？」
3. 「変容後、全体性スナップショットが記録されているか？」
4. 「このコードを読んで『なるほど』と感じるか？（特性14）」
5. 「命名は既存のユビキタス言語と共鳴しているか？（特性12）」

**レビューコメントのフォーマット**:
```
[特性N] [特性名]の観点：
[現在の状態の観察]
[変容の提案]
```

例：
```
[特性12] 共鳴（類似性）の観点：
UserAccountクラスが導入されましたが、既存コードでは同じ概念を
`account`と呼んでいます。命名を統一することで共鳴が高まります。
変容提案：`UserAccount` → `Account` に改名することを検討してください。
```

## 技術的負債の評価フレームワーク

技術的負債を「生命の質が低いセンター」として再定義する。

### 負債の重み付け評価

```
負債の重み = 生命の質の低さ × 影響範囲 × 変容コスト

生命の質の低さ: 弱い15特性の数（0〜15）
影響範囲: そのセンターが影響する他のセンターの数
変容コスト: 変容を安全に実施するのに必要な工数
```

### 負債の種類と対応する特性

| 負債の種類 | 主に欠けている特性 | 変容の優先度 |
|-----------|------------------|------------|
| ロジックの散在 | 特性2（力強いセンター） | 高 |
| 命名の混乱 | 特性12（共鳴） | 中（高コストの変容前に実施） |
| 曖昧なインターフェース | 特性3（境界） | 高（変容の前提条件になりやすい） |
| デッドコード | 特性15（不可分性） | 低コストで即実施 |
| 複雑な条件分岐 | 特性14（シンプルさ）、特性9（コントラスト） | 中 |
| God Object | 特性1（スケール）、特性2（センター）が混在 | 計画的に段階変容 |
