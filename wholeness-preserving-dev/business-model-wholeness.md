# Layer B: ビジネスモデル×顧客の全体性

## Layer B の前提

Layer B は「顧客（Customer）がコストを払う価値があるか」を問う。
Layer P（利用者が役立てているか）とは主体が異なる。

- **利用者（User）**: 実際に操作する人。Layer U・P・I の主体。
- **顧客（Customer）**: 費用を判断する人（購買担当、経営層、親企業、自分自身）。

B2C では User = Customer であることが多い。
B2B・B2B2C では乖離が生じ、この乖離が変容判断の重要な張力になる。

**設計意図の詳細 → `DESIGN-NOTES.md`（Layer B の設計意図 セクション）**

---

## Layer B のセンター構造

| センター種別 | 説明 | 弱さのシグナル |
|---|---|---|
| 顧客センター | 意思決定者・予算保持者・推進者（Champion） | 「誰が買うか」が不明確 |
| 収益センター | 価格モデル・契約形態・アップセル経路 | 収益経路が単一・脆弱 |
| 価値証明センター | ROI・KPI・成功指標 | 「価値がある」を証明できない |
| チャーンセンター | 解約のトリガー・スイッチングコスト | 解約理由が分析されていない |
| User×Customer 乖離センター | 「使う人」と「買う人」の利益相反 | 乖離が可視化されていない |

## Layer B のセンター検出プロトコル

### Step 1: 構造分析

エージェントが文脈から読める範囲でセンターを読む。

```
確認する観点:
□ 誰が費用を出しているか（Customer の特定）
□ 誰が実際に使っているか（User の特定）
□ User と Customer は同一か、乖離があるか
□ 収益モデルの形態（サブスク / 従量課金 / ライセンス / 広告 / etc.）
□ 解約・チャーンの主要トリガー（把握されているか）
□ 顧客が「価値がある」と判断する基準（ROI / KPI / 感情的価値）
```

### Step 2: B 層センターマップ（仮説）

```
## B 層センターマップ（仮説）

顧客（Customer）:
  主体: [誰か]
  購買判断の基準: [何で決めるか]
  User との関係: [同一 / 乖離あり → 乖離の内容]

収益センター:
  モデル: [形態]
  強さ: [強 / 中 / 弱]
  弱さの原因: [あれば]

価値証明センター:
  測定されているKPI/ROI: [あれば]
  証明できていない価値: [あれば]

チャーンセンター:
  主要トリガー: [把握されているか]
  潜在トリガー（仮説）: [あれば]

User×Customer 乖離:
  乖離の内容: [あれば]
  乖離が引き起こす張力: [どんな意思決定の歪みが起きているか]
```

### Step 3: 人間への問い返し — ビジネスの感覚を聞く

> 「ビジネスモデルについて、感覚として聞かせてください。
>
> **① 「これで売れているが、なんか後ろめたい」と感じることはありますか？**
>    誰かを犠牲にして成り立っている気がする、という感覚。
>
> **② 「正しいものを作っているのに、誰も払ってくれない気がする」という感覚はありますか？**
>    価値はあるはずなのに、お金の流れと噛み合っていない。
>
> **③ 「この収益モデルは、長く続けられる気がしない」と感じることはありますか？**
>    今は動いているが、どこかで壊れる気がする。
>
> **④ 「ここが変わったら、収益も体験も両方良くなる気がする」という場所はありますか？**
>    守りながら強めれば、両立できる気がする場所。
>
> 言葉にならなくてもかまいません。」

```
受け取り方:
①「後ろめたい」    → B×U/P/I の癌的張力のシグナル。誰が犠牲になっているか特定する
                   → Layer W の問いにつながる可能性がある
②「噛み合わない」  → 価値証明センターの弱さ。顧客の判断基準と提供価値のズレ
③「続かない」     → 収益・チャーンセンターの脆弱性。持続性の設計が必要
④「両立できる」   → B×C/U/P/I を同時に保存できる潜在センター
```

## B の健全性チェック（変容前に必ず確認）

```
「B が強すぎる」シグナル:
□ 直近の意思決定で Layer C（コード品質）が犠牲になったか？
□ 直近の意思決定で Layer U（操作感覚）が犠牲になったか？
□ 直近の意思決定で Layer P（利用者の実際のニーズ）が犠牲になったか？
□ 直近の意思決定で Layer I（利用者の自律・能力）が犠牲になったか？
□ Customer の要求が User のニーズを上書きしていないか？

「B が弱すぎる」シグナル:
□ 利用者は喜んでいるが、誰も払っていない
□ 価値があると信じているが、価値証明の方法がない
□ 収益経路が一本だけで、変化に脆弱

→ どちらかに「Yes」があれば、変容前に B の張力を可視化する
```

## B×他レイヤーの張力の可視化

変容提案の際、B に関係する変容は必ず以下の張力を明示する。

```
B×C の張力:
  この変容は技術負債を増やすか、減らすか？
  「速く出す」が C を侵食しないか？

B×U の張力:
  この変容はダークパターンを含まないか？
  転換率と操作感覚は同方向を向いているか？

B×P の張力:
  この変容は Customer の要求か、User のニーズか？
  乖離がある場合、どちらを優先するか、理由は何か？

B×I の張力:
  この変容は利用者の自律を支えるか、依存を強化するか？
  「自律させても払い続けたくなる構造」になっているか？
```
