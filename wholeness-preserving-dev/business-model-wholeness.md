# Layer B: ビジネスモデル×顧客の全体性

## Layer B の前提

Layer B は「顧客（Customer）がコストを払う価値があるか」を問う。
Layer P（利用者が役立てているか）とは主体が異なる。

- **利用者（User）**: 実際に操作する人。Layer U・P・I の主体。
- **顧客（Customer）**: 費用を判断する人（購買担当、経営層、親企業、自分自身）。

B2C では User = Customer であることが多い。
B2B・B2B2C では乖離が生じ、この乖離が変容判断の重要な張力になる。

**設計意図の詳細 → `DESIGN-NOTES.md`（Layer B の設計意図 セクション）**

## Layer B のセンター構造

| センター種別 | 説明 | 弱さのシグナル |
|---|---|---|
| 顧客センター | 意思決定者・予算保持者・推進者（Champion） | 「誰が買うか」が不明確 |
| 収益センター | 価格モデル・契約形態・アップセル経路 | 収益経路が単一・脆弱 |
| 価値証明センター | ROI・KPI・成功指標 | 「価値がある」を証明できない |
| チャーンセンター | 解約のトリガー・スイッチングコスト | 解約理由が分析されていない |
| User×Customer 乖離センター | 「使う人」と「買う人」の利益相反 | 乖離が可視化されていない |

## Layer B のセンター検出プロトコル

### Step 1: 構造分析

エージェントが文脈から読める範囲でセンターを読む。

```
確認する観点:
□ 誰が費用を出しているか（Customer の特定）
□ 誰が実際に使っているか（User の特定）
□ User と Customer は同一か、乖離があるか
□ 収益モデルの形態（サブスク / 従量課金 / ライセンス / 広告 / etc.）
□ 解約・チャーンの主要トリガー（把握されているか）
□ 顧客が「価値がある」と判断する基準（ROI / KPI / 感情的価値）
```

### Step 3: 人間への問い返し — ビジネスの感覚を聞く

> 「ビジネスモデルについて、感覚として聞かせてください。
>
> **① 「これで売れているが、なんか後ろめたい」と感じることはありますか？**
>    誰かを犠牲にして成り立っている気がする、という感覚。
>
> **② 「正しいものを作っているのに、誰も払ってくれない気がする」という感覚はありますか？**
>    価値はあるはずなのに、お金の流れと噛み合っていない。
>
> **③ 「この収益モデルは、長く続けられる気がしない」と感じることはありますか？**
>    今は動いているが、どこかで壊れる気がする。
>
> **④ 「ここが変わったら、収益も体験も両方良くなる気がする」という場所はありますか？**
>    守りながら強めれば、両立できる気がする場所。
>
> 言葉にならなくてもかまいません。」

```
受け取り方:
①「後ろめたい」    → B×U/P/I の癌的張力のシグナル。誰が犠牲になっているか特定する
                   → Layer W の問いにつながる可能性がある
②「噛み合わない」  → 価値証明センターの弱さ。顧客の判断基準と提供価値のズレ
③「続かない」     → 収益・チャーンセンターの脆弱性。持続性の設計が必要
④「両立できる」   → B×C/U/P/I を同時に保存できる潜在センター
```

## B の健全性チェック（変容前に必ず確認）

```
「B が強すぎる」シグナル:
□ 直近の意思決定で Layer C（コード品質）が犠牲になったか？
□ 直近の意思決定で Layer U（操作感覚）が犠牲になったか？
□ 直近の意思決定で Layer P（利用者の実際のニーズ）が犠牲になったか？
□ 直近の意思決定で Layer I（利用者の自律・能力）が犠牲になったか？
□ Customer の要求が User のニーズを上書きしていないか？

「B が弱すぎる」シグナル:
□ 利用者は喜んでいるが、誰も払っていない
□ 価値があると信じているが、価値証明の方法がない
□ 収益経路が一本だけで、変化に脆弱

→ どちらかに「Yes」があれば、変容前に B の張力を可視化する
```

## 15特性 × Layer B 対応表

| 特性 | ビジネスモデルでの現れ | 弱さの兆候 | 変容の方向 |
|------|---------------------|-----------|----------|
| 1 スケールの段階性 | 顧客セグメントに大中小の階層（個人→SMB→Enterprise）がある。価格帯・サービス内容の段階性がある | 全顧客が同一プランに押し込まれている。セグメントの意味的差異がない | セグメントの価値観の違いを基準に階層を設計する |
| 2 力強いセンター | コアバリュープロポジションが一文で言える。主要収益源が明確で揺れていない | 「何で稼いでいるか」が社内で揺れている。複数の収益仮説が競合している | 主要収益源を一本化し、他を補完として位置づける |
| 3 境界 | 「やらないこと」「担わないこと」が明確。スコープクリープを防ぐ判断基準がある | 顧客要求に引っ張られて事業領域が拡散している。「何の会社か」が外から見えない | 「この会社がすること・しないこと」を一文で書く |
| 4 交互反復 | 価値提供と価値回収（支払い）のリズムが自然に噛み合っている | 価値は提供しているのに支払いタイミングが乖離している | 顧客が「価値を感じた瞬間」と支払いのタイミングをできるだけ近づける |
| 5 正の空間 | CS・オンボーディング・更新プロセスが顧客とプロダクトの「間の空間」として機能している | 「売ったら終わり」。更新・継続フローが設計されていない | 顧客との「継続的な接触点」を設計する。契約後の価値提供プロセスを整える |
| 6 よい形 | 収益モデルが市場の力学（顧客の予算サイクル・調達プロセス）に適応している | SaaSなのに顧客が年次予算で動けない。課金単位が顧客の意思決定単位とずれている | 顧客の調達プロセス・予算管理の「形」に収益モデルを合わせる |
| 7 局所的シンメトリー | 類似セグメントへのアプローチが一貫している。同じ価値提案が同じ言語で語られる | 営業担当ごとに価値の説明が違う。同じ顧客層に別々のメッセージを送っている | 顧客セグメントごとの価値メッセージを統一する。営業・マーケ・CSで共通言語を作る |
| 8 深い相互結合 | プロダクトが顧客のワークフローに深く統合されている（正のスイッチングコスト） | 顧客が「便利」とは言うが代替ツールへの乗り換えを恐れていない | 顧客のワークフロー・データ・習慣とプロダクトを深く統合する。「一緒に育つ」関係を設計する |
| 9 コントラスト | Free/Paid・Basic/Premiumの差異が明確で価値が伝わる | 全プランの機能差が曖昧。なぜ上位プランを買うのかわからない | 各ティアの「核心的な差異」を一つに絞り込んで明示する |
| 10 グラデーション | 無料トライアル → 小プラン → 大プラン → エンタープライズへの自然な段階がある | 「小さく始めて大きく育てる」経路がない。最初から大きな契約を迫っている | 顧客が小さく始めて段階的に深くコミットできる経路を設計する |
| 11 粗っぽさ | 会社・プロダクトの個性がある。語り口が他社と交換不可能。なぜこの会社のものを使うのかが語れる | 無個性で「業界標準」に見える。なぜこの会社のものを使うのかが語れない | 会社の価値観・歴史・こだわりを言語化する。それをプロダクト・販売・CSに反映させる |
| 12 共鳴 | ブランド・価値観・語り口が全タッチポイント（LP・営業・サポート・請求書）で一致している | タッチポイントごとに別会社のような印象。マーケと営業と開発で言うことが違う | 組織全体でブランドの価値観・語り口を共有する。全タッチポイントを一貫させる |
| 13 空 | 事業の「余白」がある。次の収益源・新セグメントへの成長余地が残っている | 現収益源に過最適化されて次の手が見えない。または手をつけていない領域が多すぎる | 今の「コア」と将来の「余白」を区別する。意識的に空けておく領域を定義する |
| 14 シンプルさと静謐さ | 料金体系・契約条件が読んですぐわかる。意思決定の認知負荷が低い | 複雑な価格表。「どのプランが自分に合うか」を顧客が判断できない。例外条件が多い | 料金体系をシンプルにする。選択肢を3つ以内に絞る。例外をなくす |
| 15 不可分性 | プロダクトが顧客の業務・生活に自然に溶け込んでいる。「あって当然」の存在になっている | 顧客が解約を恐れていない。「便利だが代替できる」と思われている | 顧客のワークフローに深く統合する。「なくなったら困る」存在になることを目指す |

## B×他レイヤーの張力の可視化

変容提案の際、B に関係する変容は必ず以下の張力を明示する。

```
B×C の張力:
  この変容は技術負債を増やすか、減らすか？
  「速く出す」が C を侵食しないか？

B×U の張力:
  この変容はダークパターンを含まないか？
  転換率と操作感覚は同方向を向いているか？

B×P の張力:
  この変容は Customer の要求か、User のニーズか？
  乖離がある場合、どちらを優先するか、理由は何か？

B×I の張力:
  この変容は利用者の自律を支えるか、依存を強化するか？
  「自律させても払い続けたくなる構造」になっているか？
```
