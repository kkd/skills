# Wholeness-Preserving Development

## 7つの全体性レイヤー

```
┌──────────────────────────────────────────────────────────┐
│  Layer H: 人間×エージェントの関係の全体性                  │
│  「分析」と「感覚」が突き合わさり、変容の道筋が生成される場  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Layer W: 社会×世界の全体性（羅針盤・感覚問いのみ）  │  │
│  │              ↕ 張力 ↕                              │  │
│  │  Layer B: ビジネスモデル×顧客の全体性               │  │
│  │              ↕ 張力 ↕                              │  │
│  │  Layer I: 利用者へのインパクト（自律・依存・変化）   │  │
│  │              ↕ 張力 ↕                              │  │
│  │  Layer P: プロダクト×利用者の全体性                 │  │
│  │              ↕ 張力 ↕                              │  │
│  │  Layer U: インターフェース×人の全体性               │  │
│  │              ↕ 張力 ↕                              │  │
│  │  Layer C: コードの全体性                            │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

| Layer | 全体性の「場」 | エージェントが読む | 人間が感じ取る |
|-------|-------------|-----------------|-------------|
| C | コードベース | 依存・構造・15特性 | 「変えるのが怖い」「何か気持ち悪い」 |
| U | 人の認知・感情・行動 | UIセンター・インタラクション構造 | 「迷う」「気持ちいい」「なんか違和感」 |
| P | 利用者の生活文脈 | バックログ・ログ・記録 | 「利用者への責任感」「これは違う」 |
| I | 利用者の変化・成長・依存 | 使用パターン・自律/依存の構造 | 「使うほど頼りすぎている気がする」「育っている気がする」 |
| B | 顧客×収益構造 | 収益モデル・Customer/User 乖離 | 「後ろめたい」「噛み合っていない」「続かない気がする」 |
| W | 社会・世界・生態系 | （AI単独では読めない） | 「世界が生き生きしているか」「後ろめたさの背後」 |
| H | この対話そのもの | 全レイヤーの張力・発話パターン | 「腑に落ちた」「何かが引っかかる」 |

**コア原則**:
- エージェントは**構造分析を主に担う**（ただし人間も構造を読む）
- 人間は**身体的・倫理的・文脈的判断の最終源泉を担う**（エージェントは感覚言語を模倣するが、それは感覚ではない）

この2つを突き合わせることで、どちらも単独では見えなかった変容の道筋が生成される。

**レイヤーの目的**:
レイヤーを個別に分析することではなく、**レイヤー間の緊張の中で「何を選ぶか」の指針を得ること**だ。
特に Layer B のような既存の価値観・外部圧力が強く引っ張るとき、
自分の感じていることをどこまで信じて選択できるか——それがこのプロトコルの核心にある。
すべては仮説であり、進まなければわからない。

**自己と自我の区別**:
- **自己（じこ）**: 感じ取っている本来の自分。外部圧力の外にある感覚の源泉
- **自我（じが）**: 防衛する自分。正しくあろうとし、権威に応じ、面目を保とうとする

Layer B の圧力を感じたとき、「後ろめたい」を消したくなるのは**自我**だ。
「後ろめたい」を感じ続けているのが**自己**だ。
自我の声を自己の声と混同しないこと。この区別は、感覚を問うすべての場面で有効だ。

**Layer W は羅針盤として扱う**: 分析・診断は Layer B・I が担う。
Layer W は「後ろめたさ」感覚の背後を照らす問いとして、特定のタイミングにのみ問う。

> **レイヤーは観察モデルであり、実在の構造ではない**:
> P と I は実際にはしばしば分離不能。B は P・I を通じてしか見えないことが多い。
> H は独立レイヤーというよりメタ観測軸であり、他のすべての層に浸潤している。
> レイヤーを実体視しないこと——これは「分けて考えるための整理」であって、
> 現実がこの通りに分かれているわけではない。

## モード選択（STEP 0 の前に決定する）

### エージェントがモードを推定する基準

| リクエストの性質 | 推奨モード |
|----------------|-----------|
| バグ修正・小さなリファクタリング・命名変更 | **L: Lightweight** |
| 新機能追加・中規模リファクタリング・ストーリー検討 | **S: Standard** |
| 設計判断・アーキテクチャ変更・技術的負債の根本対処 | **D: Deep** |

エージェントはリクエストからモードを推定し、作業開始前に一言確認する:
> 「[L/S/D] モードで進めます。変更があれば教えてください。」

### 各モードのスコープ

```
L: Lightweight
  Layer C のみ分析（Layer U・P・H はスキップ）
  ※ ただしタスクが明示的に UI 変更を含む場合は Layer U を追加する
  STEP 1→2→3（1道筋のみ）→4→5
  人間への問い返し: STEP 3 終了後の1回のみ
  用途: 小さな変容を素早く実施する

S: Standard（デフォルト）
  Layer C + U（UI 変更を含む場合）+ P を分析
  全5ステップ、各ステップ末に人間への問い返し
  用途: 通常の開発作業

D: Deep
  Layer C + U + P + H を明示的に追跡
  全5ステップ + Layer H の観察を各ステップに追加
  人間への問い返し: 各ステップ末 + D コンテキスト専用の問い
  全体性スナップショット（C・U・P すべて）を必ず記録する
  用途: 設計判断・アーキテクチャ変更・UX 刷新
```

## STEP 0: コンテキスト確認（毎回最初に実行）

**Layer P の全体性へのアクセス経路**を確認する。

```
A. 利用者自身   — Layer P の全体性が直接ある。感覚の外化を支援する。
B. 開発者       — Layer P は要件を経由している。文脈の復元を試みる。
C. PO/PM       — Layer P は推測で再構成されている。乖離を可視化する。
D. 利用者 = PO  — 同一人物に2つの自己が同居している（下記参照）。
```

**コンテキスト D の特別扱い**

```
利用者-自己: 「自分がこれを使うとき、こう感じる」  [直接的・身体的]
                       ↕ 内的張力
PO-自己:    「このプロダクトとして、こうあるべき」  [判断的・未来志向]
```

危険: 利用者-自己の感覚が PO-自己の推測に無自覚に昇格される。
豊かさ: 利用者-自己が PO 判断の最も直接的な検証装置になる。

エージェントが使う問い（D コンテキスト専用、各ステップで必要に応じて）:

> 「今おっしゃったのは、どちらの自己から来ていますか？
>  A. 利用者として「自分がこれを使うとき、〜と感じる」
>  B. POとして「このプロダクトとして、〜であるべき」
>  C. 両方が重なっている → その重なりを言葉にしてみてください」

C（重なり）が最も豊かな情報を含む。

## 横断的プロトコル: スコープ拡張の検出と焦点再提案

5ステップのいずれの段階でも発動する。**ステップを中断して優先実行する。**

### 発動条件

以下のシグナルが現れたとき即座に発動する:

- 「この変更だと、〇〇にも影響が…」という連鎖が3つ以上になる
- 元のセンター（話題・変更対象）と異なるレイヤー/ドメインの話が混入し始める
- 「せっかくなので…」「ついでに…」という言葉が現れる
- 分析対象が当初の1センターから複数センターに拡散している

これらは **センターの希薄化（Center Dilution）** のシグナルだ。
焦点が失われると「最小変容」の原則が崩れ、変容は全体性を損なう。

> 広がり自体は「潜在センターの出現」である可能性がある——無視しない。
> しかし「今回の変容に含めるか」は別問題だ。

### エージェントの動き

発動したら、現在のステップを一時停止し、以下を出力する:

```
## センター再提案

当初のセンター: [元の話題・変更対象]
広がっている方向: [どのセンターが引き込まれているか]
広がりのタイプ:
  □ 上位方向（アーキテクチャ・設計全体）
  □ 下位方向（実装詳細・細部）
  □ 隣接方向（他機能・他ドメイン）

【選択肢】

A. 元のセンターを守る（スモールステップ継続）
   今回のスコープ: [当初のセンターに絞る]
   発見した影響: [別ストーリーとして記録し、後回しにする]
   リスク: [絞ることで生じる暫定対応・技術的負債]

B. センターを再定義する
   新しいセンター: [広がりを包む、より上位のセンター名]
   スコープ: [何を含め、何を含めないか]
   コスト: [当初より大きくなる変容の規模]

C. 分離する（今回 + 次回）
   今回: [最小センター] → [最小変容]
   次回: [隣接センター] → [別ストーリーとして切り出す]

発見した潜在センター（記録）:
  名前: [センター名]
  出現した文脈: [どの議論から現れたか]
  最初の一歩の仮説: [命名 / テスト / 空の構造 / ストーリー のどれか、1文]
```

**人間への問い返し:**

> 「話が[X]の方向に広がっています。
>  A・B・C のどれで進めますか？
>  感覚として、どれが『腑に落ちる』ですか？」

選択が確定したら、元のステップに戻るか、選択に応じてステップを再設定する。

## 5ステップの共同生成ワークフロー

各ステップ = **エージェントの動き** + **人間への問い返し**
両方が揃ったときそのステップは完了する。

> **ステップは往復する**: 1→2→3→4→5 は探索の足場であって、一方通行ではない。
> - 読んだ時点（STEP 1）で方向が変わることがある → STEP 0 に戻る
> - 実施中（STEP 4）に弱さの定義が変わることがある → STEP 2 に戻る
> - 感覚を聞いた瞬間（STEP 3）にセンターが入れ替わることがある → STEP 1 に戻る
>
> 前のステップに戻ることは失敗ではなく、全体性が見えてきたサインだ。

### STEP 1: 全体性を読み取る

> **L**: Layer C のみ実行（UI変更を含む場合は Layer U も）。Layer P・C×P 可視化はスキップ。
> **S/D**: Layer C + P + U（UI/UX接触面がある場合）+ C×P 可視化をすべて実行。

**エージェントの動き — Layer C:**

> **15特性の適用について**: 建築空間では直接観察できる特性も、コード・UI・
> 収益モデルへの適用は比喩的になりうる。各特性を使うとき以下を意識する:
> - **対応**: そのレイヤーで直接観察・計測できる（例: Layer C の「境界」は依存分析で見える）
> - **比喩**: 類似を示すが直接は見えない（例: Layer B の「局所的シンメトリー」）
> - **問い**: 特性を問いとして使い、思考を触発する（証明より気づきが目的）
>
> 「対応」として扱えるものは根拠を示す。「問い」として使うときはその旨を明示する。

```bash
# 依存の集中点（センター候補）
grep -r "import\|require\|from\|use " . \
  | grep -v "spec\|test\|node_modules\|.git" \
  | sed 's/.*from //' | sort | uniq -c | sort -rn | head -20

# 変更頻度ホットスポット
git log --name-only --format="" | sort | uniq -c | sort -rn | head -20

# ファイル規模
find . \( -name "*.rb" -o -name "*.py" -o -name "*.ts" -o -name "*.go" \) \
  | xargs wc -l 2>/dev/null | sort -rn | head -10
```

センター検出の詳細 → `center-analysis.md`

**エージェントの動き — Layer P:**

バックログ・Issueを構造分析し、センター仮説を立てる:
- 繰り返し登場するテーマ・ラベル（顕在センター候補）
- 長期放置Issue群（弱いセンター。変容エネルギーが届いていない）
- リオープンを繰り返すIssue（根が解決されていない潜在センター）
- 利用者が「なんとかしてきた」workaround・ログの予想外の使われ方

構造分析後、人間のモヤモヤを聞いて補正する:
> 「バックログを見ていて感覚として——
>  ① 見るたびに『これ、なんか気になる』と感じるIssueやテーマはありますか？
>  ② 『ずっと先送りにしてしまっている』ものはありますか？
>  ③ 『本当はここが一番大事なのに、バックログに入っていない』と感じることはありますか？
>  言葉にならなくてもかまいません。」

詳細 → `center-analysis.md`（Layer P セクション）、`product-wholeness.md`

**エージェントの動き — Layer U（UI/UX接触面がある場合）:**

AIが構造から読める範囲を分析する（仮説として提示）:
- 多くの画面から参照されるコンポーネント（構造的センター）
- ルーティング分岐点（フローセンター）
- エラー・ローディング状態の集中点（弱さのセンター候補）

AIが単独では検出できない部分を人間の感覚で補う:
> 「このUI・画面・操作について感覚として——
>  ① 使うたびに『引っかかる』『ここで止まる』場所はありますか？
>  ② 『壊れそう』『触りたくない』と感じる画面・操作はありますか？
>  ③ 『このUIのここが好き』『ここは残したい』と感じる部分はありますか？
>  ④ 『こうなったら気持ちよく使える気がする』という感覚はありますか？」

詳細 → `center-analysis.md`（Layer U セクション）、`ui-ux-wholeness.md`

**エージェントの動き — Layer H:**

会話そのものを観察し、エージェントが読み取った内容を開示する:
- 同じ概念を言い直した・修正した箇所（共有できていないセンター）
- 「それじゃない」と返された箇所（ズレのセンター）
- 話が自然に広がったトピック（共鳴しているセンター）
- 言いかけて止まった・曖昧な言葉を使い続けた箇所（潜在センター）

> **Layer H 観察時の必須制約**:
> - これはすべて**仮説**として提示する。断定しない。「〜のように見えます」「〜かもしれません」
> - 「当てにいく」ことを禁止する。LLM のもっともらしい物語化・心理読みに滑らないこと
> - 人間が「違う」と言ったら、即座に捨てる。食い下がらない
> - Layer H の観察を「正解」として扱うのは最も危険な使い方だ

詳細 → `center-analysis.md`（Layer H セクション）

**エージェントの動き — Layer I（利用者インパクト接触面がある場合）:**

利用者の変化・自律・依存に関わるタスクのとき実行:
- 使用パターンから「使うほど自律するか、依存するか」を読む
- B×I 張力: 解約防止のために依存が設計されていないかを確認

人間の感覚で補う:
> 「使い続けた利用者について感覚として——
>  ① 使うほど利用者が自分でできることが増えている気がする部分はありますか？
>  ② 使うほど利用者がこれなしでは動けなくなっている気がする部分はありますか？
>     それは愛着からですか？それとも囲い込みからですか？」

詳細 → `impact-wholeness.md`（Layer I セクション）

**エージェントの動き — Layer B（ビジネスモデル接触面がある場合）:**

設計判断・機能優先度・顧客要求に関わるタスクのとき実行:
- Customer と User が同一か確認する
- B の健全性チェック（強すぎ・弱すぎ）を実施

人間の感覚で補う:
> 「ビジネスモデルについて感覚として——
>  ① 売れているが後ろめたいことはありますか？
>  ② 価値があるのに噛み合っていない感覚はありますか？
>  ③ この収益モデル、長く続けられる気がしますか？」

詳細 → `business-model-wholeness.md`

**Layer W（大きな設計判断・「後ろめたさ」が出たとき・D モードの節目のみ）:**

> 「このプロダクトが10年後に広く使われている世界を想像したとき、どう感じますか？
>  より生き生きしている感じがしますか？それとも何か後ろめたいですか？」

「後ろめたい」が来たら Layer B・I のどこに起因するかを掘り下げる。

詳細 → `impact-wholeness.md`（Layer W セクション）

**全体性を可視化:**

```
## 全体性スナップショット（現在）

Layer C 主要センター: [名前・強さ・関連特性]
Layer U 主要センター: [構造分析仮説 / 感覚入力による補正]（UI接触面がある場合）
Layer P 主要センター: [Issue分析仮説 / モヤモヤ感覚による補正]
Layer I 観察: [自律センター / 依存センター（愛着 or 囲い込み）]（接触面がある場合）
Layer B 観察: [Customer/User 乖離 / 収益センター強さ / 後ろめたさシグナル]（接触面がある場合）
Layer W 感覚: [生き生き / 後ろめたい / わからない]（大判断・節目のみ）
Layer H 観察:
  共有できていない概念: [名前]
  共鳴したトピック: [名前]
  言語化されていない潜在センター: [仮名]

複数レイヤーに共通して現れるセンター:
  - [センター名]: [どのレイヤーに現れているか]
  共通の根の仮説: [全体性の核心に近い可能性]
```

**人間への問い返し:**

> 「この全体性の読み取りを見て、あなたが今感じていることと重なりますか？
>  数字や構造には現れていないが、感じていることはありますか？」

*D コンテキストの場合、さらに:*
> 「今感じていることは、どちらの自己から来ていますか？」

**人間が返答するまで STEP 2 に進まない。**

### STEP 2: もっとも弱い部分を特定する

> **L**: Layer C のみ診断。弱い特性は上位1つに絞る。Layer P・U の弱さはスキップ（UI変更含む場合は U も上位1つ）。
> **S**: Layer C + P + U（UI接触面あり）、上位3つ。
> **D**: Layer C + P + U、上位3つ + Layer H の観察（対話の全体性が失われるシグナルを確認）。

**エージェントの動き:**

Layer C の弱さ（15特性で診断 → `fifteen-properties.md`）:
```
最も弱い特性（上位3つ / L モードは上位1つ）:
  1. 特性[ ]: [コード上の具体的な現れ]
  2. 特性[ ]: [...]
  3. 特性[ ]: [...]
```

Layer U の弱さ（UI/UX接触面がある場合）:

| 利用者の言動 | 弱いセンター | 対応特性 |
|------------|-----------|--------|
| 「どこを押せばいい？」 | 焦点センター | 特性2・9 |
| 操作後に何が起きたかわからない | インタラクションセンター | 特性4・8 |
| 「なんか違和感がある」 | 局所的シンメトリー/共鳴 | 特性7・12 |
| ツールチップへの依存 | よい形・不可分性 | 特性6・15 |

詳細 → `ui-ux-wholeness.md`

Layer P の弱さ:

| シグナル | 弱いセンター |
|---------|------------|
| 「使えるけど好きじゃない」 | インタラクションセンター |
| 「何のプロダクトか」が薄れる | コアバリューセンター（特性2） |
| workaround が多い | 潜在センターが顕在化していない |

**C×P 共通の根を探す:**

```
例:
  C: 同じロジックが3箇所に散在（特性12「共鳴」の欠如）
  P: 新機能の追加が「どこに書けばいいか」で毎回止まる
     ↓ 共通の根: ドメインの中心概念が強いセンターとして存在していない（特性2）
```

**人間への問い返し:**

> 「C側では[●]、P側では[▲]を弱さとして読みましたが、
>  あなたが一番『そうなんだよ』と感じるのはどれですか？
>  私が読めていない弱さがありますか？」

感覚が「それじゃない」→ 分析を見直す。
感覚が「それだ」→ その感覚に乗る。
感覚が揺れている → 揺れ自体が情報。もう一層掘る。

*D コンテキスト:*
> 「利用者として感じる弱さと、POとして感じる弱さが違いますか？」

**特定できたら、アプローチを確認する:**

弱さへの対処は「強める」だけではない。
特定したセンターに対して、どちらで進むかを確認する：

> 「[特定したセンター]について——
>  A. 直接強化する（そのセンター自体を変容する）
>  B. 補完する（そのセンターは今のままで、周囲のセンターで支える）
>
>  どちらが今のプロジェクトに合っていますか？」

```
A「強化する」を選んだとき → STEP 3 で直接変容の道筋を探る
B「補完する」を選んだとき → STEP 3 で周囲のセンターから支える変容を探る
どちらか迷う           → 「まず補完で始めて、次のサイクルで強化を検討する」が安全
```

### STEP 3: 次の変容を共に見出す

> **L**: 3レベル探索はスキップ。STEP 2 で特定した弱さに直結する道筋を1つだけ提案し、最初の一歩を即提示する。人間への問い返しはここで1回行い確認を取る。
> **S**: 3レベル探索 + 3道筋 + 最初の一歩の具体化。
> **D**: S に加え、D コンテキスト検証（2つの自己）を実施。

**エージェントの動き — 潜在センターの3レベル探索（C+P+U 統合）:**

```
上位方向:
  C: 上位モジュールが期待しているが、まだない「何か」
  U: 利用者の認知の流れが期待しているが、画面にない「手がかり」（UI接触面あり）
  P: 利用者の生活文脈の中で奉仕できていない部分
  → 共通の潜在センター: [複数層に現れるもの]

下位方向:
  C: このコードの中に名前が付いていない概念
  U: インタラクションの中に顕在化すべきフィードバックパターン（UI接触面あり）
  P: 利用者が「なんとなく一緒にやる」操作の中に顕在化すべきもの
  → 共通の潜在センター: [複数層に現れるもの]

隣接方向:
  C: 隣のモジュールが持つ特性で、このモジュールが欠いているもの
  U: 隣の画面/フローが持つ操作感で、この画面が欠いているもの（UI接触面あり）
  P: 利用者がプロダクトと一緒に使うツール・習慣・関係者
  → 共通の潜在センター: [複数層に現れるもの]
```

**C×P×U 張力への道筋（アプローチ選択に応じて分岐）:**

STEP 2 で選んだアプローチに応じて探索する。

```
【A: 強化する】弱いセンター自体を変容する

道筋A-1: P優先（フィーチャ先行）
  Layer P への影響: / Layer C への影響: / Layer U への影響:
  保存される全体性: / 失われる全体性:
  最初の1手: [下記「最初の一歩の設計」で具体化する]

道筋A-2: C優先（抽象化先行）
  [同上]

道筋A-3: 統合（C+P+U を同時に保存する最小変容）
  着眼点: [張力の共通の根に働きかける]
  [同上] / コスト・リスク:
```

```
【B: 補完する】弱いセンターはそのままで、周囲のセンターで支える

問い: 「このセンターの弱さを、隣のどのセンターが補えるか？」

補完の方向:
  上位センターで支える: [上位センターを強化して弱いセンターをカバーする]
  隣接センターで支える: [隣のセンターにその役割を持たせる]
  テスト・監視で支える: [弱さを可視化・早期検知して被害を局所化する]

最初の1手: [変容は弱いセンターではなく、補完するセンター側に入れる]
コスト比較: 強化と比べて小さな変容で済む場合、Bを推奨する
```

道筋が見つからないときは正直にそう伝える。

**道筋ごとに「最初の一歩」を必ず具体化する:**

最初の一歩は以下の優先順で選ぶ。上位ほど小さく、可逆性が高い。

```
優先1: 命名する
  潜在センターに名前をつける。コメント・定数・型定義・空のクラス名のどれか。
  例: // TODO: OrderFulfillment センターをここに抽出する

優先2: テストで記述する（TDD）
  そのセンターが存在するなら通るはずの failing test を1つ書く。
  例: test("OrderFulfillment は在庫確認と配送指示を持つ") { ... }

優先3: 空の構造を作る
  名前だけのモジュール/クラス/ファイルを作り「場所を確保する」。
  実装ゼロでよい。既存テストが全通過することを確認する。

優先4: ストーリーとして書く（P層が先行する場合）
  最小のユーザーストーリー形式で潜在センターを記述する。
  コードは書かない。
```

```
## 最初の一歩の提案

選択した道筋: [A / B / C]
特定した潜在センター: [名前]

最初の一歩:
  種別: [命名 / テスト / 空の構造 / ストーリー]
  具体的なアクション: [1文で書ける粒度]
  完了の確認: [これができたら「一歩が踏まれた」と判断できること]
  可逆性: [元に戻す方法]

構造保存の根拠:
  強化される特性: [15特性の番号と名前]
  損なわれる可能性がある全体性: [なければ「なし」]
```

**人間への問い返し — 感覚マッピング:**

道筋を提示する前に、まず感覚を聞く。エージェントの分析と人間の感覚を重ねてから道筋を決める。

> 「少し立ち止まって、感覚として聞かせてください。
>
> **① 「なんか違う」「読むたびに引っかかる」のはどこですか？**
>    使うたびに、見るたびに、ちょっと止まる場所。
>
> **② 「触ったら壊れそう」「怖い」と感じるのはどこですか？**
>    近づきたくない、そっとしておきたい——恐れに近い感覚。
>
> **③ 「大事にしたい」「ここは残したい」のはどこですか？**
>    愛着がある、これが活きているから全体が成り立っている気がする。
>
> **④ 「ここが育ったら気持ちいい」と感じる部分はありますか？**
>    まだそこにはないけど、変わったらしっくりくる気がする。
>
> 言葉にならなくてもかまいません。
> 『あのへん』『なんか全体的に』でも受け取ります。」

> **感覚についての注意**: 感覚を丁寧に聞くことは、正しさを保証しない。
> 感覚は後付けの理由づけ・その場の気分・権威ある人の好み・
> 実際の使用から切れた印象論になりうる。
> 感覚を受け取ったあと、「再観察」「小さな実験」「実際の使用場面への引き戻し」
> のいずれかを経ることで、感覚に根拠の重みを加える。
> 感覚は重要な情報だが、感覚そのままを真理として扱わない。

**感覚を受け取ったら、道筋の設計に反映する:**

```
①「引っかかる」      → 弱いセンター・変容候補として道筋に組み込む
②「触ったら壊れそう」→ 脆いセンター。変容前にテストで保護する。後回しを検討する
③「大事にしたい」    → 生きているセンター。変容の制約条件として最初から設計に組み込む
④「育てたら気持ちいい」→ 潜在センターの手がかり。道筋の着眼点にする
```

②と③はどちらも「守る」だが意味が違う:
- ②は恐れ起因 → 補強・保護が先。変容の対象にするなら慎重に
- ③は愛着起因 → 変容で失ってはならないもの。道筋の成否判断基準になる

感覚が「特にない」→ エージェント側の分析仮説を先に提示し、どれかに引っかかりがあるか聞く。
感覚が揺れている → 揺れ自体が情報。「そのあたり」を起点に道筋を設計する。

**感覚マッピングを踏まえて道筋を提示し、確認する:**

> 「この感覚を踏まえると——
>  [道筋X] が一番しっくりきそうです。
>  『それは違う』という感覚はありますか？」

道筋が決まったら、最初の一歩の提案を出し、STEP 4 に進む前に確認を取る。

> **L**: ①〜④を1問にまとめる:「今回、引っかかる・壊れそう・残したい・育てたい、で一番近いのはどれですか？」感覚を受け取ったら即道筋を提示する。
> **D**: ①〜④に加え、レイヤーを指定して深掘りする。

*D コンテキスト — 道筋を2つの自己で検証:*

> 「道筋[X]を選ぶとして——
>  利用者として: 自分が実際に使う場面を想像したとき、どう感じますか？
>  POとして: これをシップするとき、どう感じますか？
>  2つの感覚は一致していますか？ずれていますか？」

ずれは道筋の誤りではない。まだ見えていない全体性のシグナルだ。

### STEP 4: 最小変容を実施する

> **L**: テスト実行と依存変化確認のみ。ユーザーストーリー記述はスキップ。実施中の問い返しは予期しない事態が起きたときのみ。
> **S**: 実施原則に従い実行。予期しない事態で問い返し。
> **D**: S に加え、実施中も D コンテキスト検証を使う。ユーザーストーリー記述を必ず行う。

**実施原則（C+P+U 統合）:**

1. 1ステップ = テストが通り、全体性の変化を感じられる最小単位
2. C の変容: 強化する15特性を明示する
3. U の変容（UI接触面あり）: 「利用者の操作感覚は変わったか」を問う
4. P の変容: 「潜在センターの顕在化」として記述する
5. 各ステップ後: C+P+U すべての全体性が保たれているか感じ取る

```bash
# テスト実行（各ステップ後に必ず）
<your-test-command>

# 依存変化の確認
git diff --stat HEAD
```

**ユーザーストーリーの記述（構造保存変容版）:**

```
## Story: [タイトル]
潜在センターの記述:
  利用者の文脈: / 現在の弱さ: / 潜在センター:
構造保存の判定:
  □ コアバリューセンターを強化する
  □ 利用者文脈との不可分性を保つ（特性15）
  □ 境界センターを尊重する（特性3）
  □ スモールステップで実施できる / 可逆である
強化する15特性: [番号と名前]
完了の定義:
  □ C: テスト全通過
  □ U: 操作感覚が「気づいたら変わっていた（でも自然）」（UI接触面あり）
  □ P: 利用者の文脈の中に自然に収まっている
  □ C+P+U: 全体性スナップショットを記録した
```

**実施中の問い返し:**
予期しない状況が起きたとき即座に問う:
> 「[状況]が起きています。続けるか、立ち止まるか、一緒に判断したいです。」

*D コンテキスト — 実装中に使える検証:*
> 「今実装しているものを、自分が利用者として使う場面を想像してください。
>  何か引っかかりはありますか？」

### STEP 5: 変容をテストし、元に戻すか決める

> **L**: テスト全通過を確認し、「感覚」の問い返しを1文で行う。スナップショット記録はスキップ。
> **S**: 成功判断表 + 感覚の問い返し。
> **D**: S に加え、全体性スナップショットを必ず記録。D コンテキストの最終確認を行う。

**成功の判断:**

> **注**: 以下の成功条件は「よい振り返りを起こすための問いの系」として機能する。
> 複数人で使う場合は解釈が割れる。主観指標は方向として使い、観察可能指標で補う。

| Layer | 主観指標（振り返りの問い） | 観察可能指標 |
|-------|--------------------------|------------|
| C | 弱かった特性が強まった感覚 | テスト全通過 / 変更行数が意図した範囲内 / 新たな循環依存が生まれていない |
| U | 「気づいたら変わっていた（でも自然）」 | 「あれ、どこ行った？」と感じる操作がない / 変更前後の操作ステップ数 |
| P | 利用者文脈に自然に収まる感覚 | コアバリューとの接続が説明できる / 既存 workaround が不要になった |
| C+P+U | すべての全体性が保たれた感覚 | 一方を保存して他方が損なわれていない（チェックリストで確認）|

> **元に戻す候補のシグナル**: 「別の場所に新たな弱さが生まれた」
> 「一方を保存したが他方が損なわれた」——感覚と観察可能指標の両方で確認する。

**人間への問い返し:**

> 「テスト結果とは別に、
>  このコードベース/プロダクトを今どう感じていますか？」

感覚が「よくなった」→ 進む。
感覚が「何かおかしい」→ テストが通っていても立ち止まる。
**感覚と構造が一致するまで次のステップに進まない。**

> **現実からのフィードバックが最重要**:
> 事前に考えたこと・感じたことは仮説だ。
> 実装して、使って、現実が返してきたものを最も重く扱う。
> STEP 1〜4 の分析・感覚マッピングに縛られず、
> 現実が「違う」と言ったら、分析を捨てて現実に従う。

*D コンテキストの最終確認:*

> 「利用者として: 自分が使いたいと思いますか？
>  POとして: これをシップしたいと思いますか？
>  どちらかが『何か引っかかる』なら、一緒に掘り下げましょう。」

## Layer H の構造保存：対話の全体性を守る

> **Layer H 全体への制約**: 本当に潜在センターを見ているのか、単に会話パターンを
> 物語化しているだけかの区別が難しい。常に「仮説」として扱い、
> 人間が否定したら即捨てる構えを保つこと。

**失われるシグナル:**

| シグナル | 何が損なわれているか |
|---------|------------------|
| 提案を検討なく受け入れ続ける | 感覚センターが機能していない |
| 感覚と分析が**常に一致する** | 突き合わせが起きていない |
| 2つの自己の区別が消える（D） | 融合が起き、検証装置が失われている |

**対話の全体性を保つ原則:**

1. 感覚を先に問う場合もある — 分析の前に「今何を感じているか」を聞く
2. 「違う」を歓迎する — 感覚が分析と違うとき、感覚を材料に掘り下げる
3. ずれを消さない — C×P の張力も、D の2つの自己のずれも、解消すべき問題ではなく変容の材料だ
4. コンテキストは変わりうる — 対話の中で人間のコンテキストが変化したら更新する
