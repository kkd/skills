# Wholeness-Preserving Development

## 3つの全体性レイヤー

```
┌──────────────────────────────────────────────────────────┐
│  Layer H: 人間×エージェントの関係の全体性                  │
│  「分析」と「感覚」が突き合わさり、変容の道筋が生成される場  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Layer P: プロダクト×利用者の全体性                 │  │
│  │              ↕ 張力 ↕                              │  │
│  │  Layer C: コードの全体性                            │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

| Layer | 全体性の「場」 | エージェントが読む | 人間が感じ取る |
|-------|-------------|-----------------|-------------|
| C | コードベース | 依存・構造・15特性 | 「変えるのが怖い」「何か気持ち悪い」 |
| P | 利用者の生活文脈 | バックログ・ログ・記録 | 「利用者への責任感」「これは違う」 |
| H | この対話そのもの | C×P の張力・構造 | 「腑に落ちた」「何かが引っかかる」 |

**コア原則**: エージェントは構造を読む。人間は全体性を感じ取る。
この2つを突き合わせることで、どちらも単独では見えなかった変容の道筋が生成される。

詳細リファレンス:
- 15特性 × コード対応 → `references/fifteen-properties.md`
- センター検出プロトコル → `references/center-analysis.md`
- プロダクト全体性 → `references/product-wholeness.md`
- アジャイル/XP 統合 → `references/agile-xp-integration.md`

---

## STEP 0: コンテキスト確認（毎回最初に実行）

**Layer P の全体性へのアクセス経路**を確認する。

```
A. 利用者自身   — Layer P の全体性が直接ある。感覚の外化を支援する。
B. 開発者       — Layer P は要件を経由している。文脈の復元を試みる。
C. PO/PM       — Layer P は推測で再構成されている。乖離を可視化する。
D. 利用者 = PO  — 同一人物に2つの自己が同居している（下記参照）。
```

**コンテキスト D の特別扱い**

```
利用者-自己: 「自分がこれを使うとき、こう感じる」  [直接的・身体的]
                       ↕ 内的張力
PO-自己:    「このプロダクトとして、こうあるべき」  [判断的・未来志向]
```

危険: 利用者-自己の感覚が PO-自己の推測に無自覚に昇格される。
豊かさ: 利用者-自己が PO 判断の最も直接的な検証装置になる。

エージェントが使う問い（D コンテキスト専用、各ステップで必要に応じて）:

> 「今おっしゃったのは、どちらの自己から来ていますか？
>  A. 利用者として「自分がこれを使うとき、〜と感じる」
>  B. POとして「このプロダクトとして、〜であるべき」
>  C. 両方が重なっている → その重なりを言葉にしてみてください」

C（重なり）が最も豊かな情報を含む。

## 5ステップの共同生成ワークフロー

各ステップ = **エージェントの動き** + **人間への問い返し**
両方が揃ったときそのステップは完了する。

---

### STEP 1: 全体性を読み取る

**エージェントの動き — Layer C:**

```bash
# 依存の集中点（センター候補）
grep -r "import\|require\|from\|use " . \
  | grep -v "spec\|test\|node_modules\|.git" \
  | sed 's/.*from //' | sort | uniq -c | sort -rn | head -20

# 変更頻度ホットスポット
git log --name-only --format="" | sort | uniq -c | sort -rn | head -20

# ファイル規模
find . \( -name "*.rb" -o -name "*.py" -o -name "*.ts" -o -name "*.go" \) \
  | xargs wc -l 2>/dev/null | sort -rn | head -10
```

センター検出の詳細 → `references/center-analysis.md`

**エージェントの動き — Layer P:**

インタビュー記録・バックログ・ログから整理:
- 利用者が「なんとかしてきた」workaround
- センターが不明確なバックログストーリー
- ログの中の「予想外の使われ方」（= 利用者が作った非公式センター）

詳細 → `references/product-wholeness.md`

**C×P の張力を可視化:**

```
## 全体性スナップショット（現在）
Layer C 主要センター: [名前・強さ・関連特性]
Layer P 主要センター: [名前・利用者との関係]
C×P の接触点（張力）:
  P側: [何をしたいか・何が弱いか]
  C側: [何が必要か・何が障壁か]
  共通の根の仮説: [両者が同じ全体性に属する可能性]
```

**人間への問い返し:**

> 「この全体性の読み取りを見て、あなたが今感じていることと重なりますか？
>  数字や構造には現れていないが、感じていることはありますか？」

*D コンテキストの場合、さらに:*
> 「今感じていることは、どちらの自己から来ていますか？」

**人間が返答するまで STEP 2 に進まない。**

---

### STEP 2: もっとも弱い部分を特定する

**エージェントの動き:**

Layer C の弱さ（15特性で診断 → `references/fifteen-properties.md`）:
```
最も弱い特性（上位3つ）:
  1. 特性[ ]: [コード上の具体的な現れ]
  2. 特性[ ]: [...]
  3. 特性[ ]: [...]
```

Layer P の弱さ:

| シグナル | 弱いセンター |
|---------|------------|
| 「使えるけど好きじゃない」 | インタラクションセンター |
| 「何のプロダクトか」が薄れる | コアバリューセンター（特性2） |
| workaround が多い | 潜在センターが顕在化していない |

**C×P 共通の根を探す:**

```
例:
  C: 同じロジックが3箇所に散在（特性12「共鳴」の欠如）
  P: 新機能の追加が「どこに書けばいいか」で毎回止まる
     ↓ 共通の根: ドメインの中心概念が強いセンターとして存在していない（特性2）
```

**人間への問い返し:**

> 「C側では[●]、P側では[▲]を弱さとして読みましたが、
>  あなたが一番『そうなんだよ』と感じるのはどれですか？
>  私が読めていない弱さがありますか？」

感覚が「それじゃない」→ 分析を見直す。
感覚が「それだ」→ その感覚に乗る。
感覚が揺れている → 揺れ自体が情報。もう一層掘る。

*D コンテキスト:*
> 「利用者として感じる弱さと、POとして感じる弱さが違いますか？」

---

### STEP 3: 次の変容を共に見出す

**エージェントの動き — 潜在センターの3レベル探索（C+P 統合）:**

```
上位方向:
  C: 上位モジュールが期待しているが、まだない「何か」
  P: 利用者の生活文脈の中で奉仕できていない部分
  → 共通の潜在センター: [両者に現れるもの]

下位方向:
  C: このコードの中に名前が付いていない概念
  P: 利用者が「なんとなく一緒にやる」操作の中に顕在化すべきもの
  → 共通の潜在センター: [両者に現れるもの]

隣接方向:
  C: 隣のモジュールが持つ特性で、このモジュールが欠いているもの
  P: 利用者がプロダクトと一緒に使うツール・習慣・関係者
  → 共通の潜在センター: [両者に現れるもの]
```

**C×P 張力への3つの道筋:**

```
道筋A: P優先（フィーチャ先行）
  Layer P への影響: / Layer C への影響:
  保存される全体性: / 失われる全体性:
  最初の1手:

道筋B: C優先（抽象化先行）
  [同上]

道筋C: 統合（C+P を同時に保存する最小変容）
  着眼点: [張力の共通の根に働きかける]
  [同上] / コスト・リスク:
```

道筋 C が見つからないときは正直にそう伝える。

**人間への問い返し:**

> 「3つの道筋を提示しました。
>  『腑に落ちる』感覚があるのはどれですか？
>  『それは違う』という感覚はありますか？
>  感覚の理由は言葉にならなくてもかまいません。」

*D コンテキスト — 道筋を2つの自己で検証:*

> 「道筋[X]を選ぶとして——
>  利用者として: 自分が実際に使う場面を想像したとき、どう感じますか？
>  POとして: これをシップするとき、どう感じますか？
>  2つの感覚は一致していますか？ずれていますか？」

ずれは道筋の誤りではない。まだ見えていない全体性のシグナルだ。

---

### STEP 4: 最小変容を実施する

**実施原則（C+P 統合）:**

1. 1ステップ = テストが通り、全体性の変化を感じられる最小単位
2. C の変容: 強化する15特性を明示する
3. P の変容: 「潜在センターの顕在化」として記述する
4. 各ステップ後: C+P 両方の全体性が保たれているか感じ取る

```bash
# テスト実行（各ステップ後に必ず）
<your-test-command>

# 依存変化の確認
git diff --stat HEAD
```

**ユーザーストーリーの記述（構造保存変容版）:**

```
## Story: [タイトル]
潜在センターの記述:
  利用者の文脈: / 現在の弱さ: / 潜在センター:
構造保存の判定:
  □ コアバリューセンターを強化する
  □ 利用者文脈との不可分性を保つ（特性15）
  □ 境界センターを尊重する（特性3）
  □ スモールステップで実施できる / 可逆である
強化する15特性: [番号と名前]
完了の定義:
  □ C: テスト全通過
  □ P: 利用者の文脈の中に自然に収まっている
  □ C+P: 全体性スナップショットを記録した
```

**実施中の問い返し:**
予期しない状況が起きたとき即座に問う:
> 「[状況]が起きています。続けるか、立ち止まるか、一緒に判断したいです。」

*D コンテキスト — 実装中に使える検証:*
> 「今実装しているものを、自分が利用者として使う場面を想像してください。
>  何か引っかかりはありますか？」

---

### STEP 5: 変容をテストし、元に戻すか決める

**成功の判断:**

| Layer | 成功 | 元に戻す候補 |
|-------|------|------------|
| C | テスト全通過 / 弱かった特性が強まった | 別の場所に新たな弱さが生まれた |
| P | 利用者文脈に自然に収まる / コアが強まった | 機能が使われるがコアと結びついていない |
| C+P | 両方の全体性が保たれた | 一方を保存したが他方が損なわれた |

**人間への問い返し:**

> 「テスト結果とは別に、
>  このコードベース/プロダクトを今どう感じていますか？」

感覚が「よくなった」→ 進む。
感覚が「何かおかしい」→ テストが通っていても立ち止まる。
**感覚と構造が一致するまで次のステップに進まない。**

*D コンテキストの最終確認:*

> 「利用者として: 自分が使いたいと思いますか？
>  POとして: これをシップしたいと思いますか？
>  どちらかが『何か引っかかる』なら、一緒に掘り下げましょう。」

## Layer H の構造保存：対話の全体性を守る

**失われるシグナル:**

| シグナル | 何が損なわれているか |
|---------|------------------|
| 提案を検討なく受け入れ続ける | 感覚センターが機能していない |
| 感覚と分析が**常に一致する** | 突き合わせが起きていない |
| 2つの自己の区別が消える（D） | 融合が起き、検証装置が失われている |

**対話の全体性を保つ原則:**

1. 感覚を先に問う場合もある — 分析の前に「今何を感じているか」を聞く
2. 「違う」を歓迎する — 感覚が分析と違うとき、感覚を材料に掘り下げる
3. ずれを消さない — C×P の張力も、D の2つの自己のずれも、解消すべき問題ではなく変容の材料だ
4. コンテキストは変わりうる — 対話の中で人間のコンテキストが変化したら更新する
