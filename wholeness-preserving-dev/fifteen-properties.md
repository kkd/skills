# アレグザンダーの15の生命特性 × コード対応

> アレグザンダーが長年の観察を通じて特定した、生きているシステムに繰り返し
> 現れる15の特性。変容の各ステップで「どの特性を強化するか」を意識的に選ぶ。

---

## 読み方

各特性について以下を記述する：
- **コードでの現れ**: この特性が強いコードの具体的な特徴
- **弱さの兆候**: この特性が欠けているときに現れるシグナル
- **変容の方向**: この特性を強化するための代表的なアプローチ

診断では全15を確認し、「強い/弱い」を記録する。
**変容の方向は「もっとも弱い特性を強化する」方向に設計する**。

---

## 特性1：スケールの段階性（Levels of Scale）

**意味**: センターは大→中→小の段階的な大きさで存在する。
約1:2〜1:4のサイズ比が自然界では一般的。

**コードでの現れ**:
- システム → モジュール → クラス → メソッド の粒度が自然な階層をなしている
- 各レベルが適切な責務を持ち、レベルをまたがない処理がない
- ディレクトリ構造が意味的な階層を表現している

**弱さの兆候**:
- 全部が同じ粒度に見える（モノリシックな平坦構造）
- 1つのクラスが数百〜数千行になっている
- ネストが深すぎる（10段以上）または浅すぎる（1段のみ）

**変容の方向**:
巨大なクラスを適切なサブセンターに分解する。
適切な抽象階層を導入し、大中小の粒度をそろえる。

---

## 特性2：力強いセンター（Strong Centers）

**意味**: 全体の中で、他のセンターを組織化する核となる焦点が存在する。
力強いセンターは明確な内部構造を持ち、重要なプロセスと結びついている。

**コードでの現れ**:
- ドメインの中心概念が明確なクラス/モジュールとして体現されている
- コアロジックが散在せず、一箇所に集約されている
- 参照の集中点が意味的に重要な箇所と一致している

**弱さの兆候**:
- 「このロジックはどこに書くべきか」が自明でない
- 似たロジックが複数の場所に散らばっている
- God Objectとして肥大化している（全体性ではなく責務の混在）

**検出**:
```bash
grep -r "import\|require" . | sed 's/.*from //' | \
  sort | uniq -c | sort -rn | head -10
```

**変容の方向**:
散在したロジックをドメインセンターに集約する。
God Objectは「力強いセンターの欠如」の結果であり、
先に弱いセンターを見つけて分離する。

---

## 特性3：境界（Boundaries）

**意味**: センターは内側の全体性を強化し、かつ内部と外部を接続する明確な境界を持つ。
境界はセンターの大きさに匹敵するくらいの「厚さ」を持つことが多い。

**コードでの現れ**:
- モジュール/パッケージのインターフェースが明示的に定義されている
- 外部に公開するものと内部に留めるものが明確に分かれている
- 境界を越える通信は意図的なAPIを通じてのみ行われる

**弱さの兆候**:
- 暗黙のグローバル状態がある
- パッケージ/モジュールをまたぐ直接参照がある
- 「どこまでが内部か」が読んでもわからない

**変容の方向**:
インターフェースを明示化し、内部実装を隠蔽する。
パッケージ境界にファサードを置く。

---

## 特性4：交互反復（Alternating Repetition）

**意味**: 単純な繰り返しよりも、2種類の要素が交互にリズミカルに現れるとき、
システムは生き生きとする。

**コードでの現れ**:
- データ定義と操作の交互配置
- 抽象インターフェースと具体実装の対応
- テストコードと実装コードのリズミカルな対応（TDD）
- ドメイン層とインフラ層の明確な交互構造

**弱さの兆候**:
- データクラスと処理クラスが混在して区別できない
- テストの粒度が実装と対応していない
- 抽象と具体の境が曖昧

**変容の方向**:
CQS（コマンドとクエリの分離）を適用する。
テストと実装を対応させる構造を意識する。

---

## 特性5：正の空間（Positive Space / 相補性）

**意味**: センター同士が利用可能なスペースを埋めるように適応し合い、
その間の空間自体がポジティブで意味あるものになる。

**コードでの現れ**:
- クラス間の「隙間」（通信プロトコル、インターフェース）が意味を持つ
- モジュール間の境界空間（アダプター、ゲートウェイ）が整然としている
- 責務の重複がなく、かつカバーされない責務もない

**弱さの兆候**:
- モジュール間の通信が場当たり的
- 「どのクラスの責任か」が判断できない空白地帯がある
- アダプター層が肥大化している

**変容の方向**:
責務の境界を見直し、センター間の「間の空間」を整える。

---

## 特性6：よい形（Good Shape / 力への適応）

**意味**: よい形は、局所的に働く力に最も適応した、シンプルな形から生まれる。
コンパクトであることから一体感と完全性が感じられる。

**コードでの現れ**:
- 関数/クラスがそのドメインの「形」を自然に反映している
- データ構造がドメインの力学（制約・不変条件）に適応している
- アルゴリズムがその問題の性質に最も適した形をしている

**弱さの兆候**:
- 汎用的すぎて、特定のドメインの形が見えない
- データ構造がDBスキーマやAPIの都合を反映しすぎている
- アルゴリズムが問題の性質と乖離している

**変容の方向**:
ドメインの力学（制約・不変条件）をコードの形に反映させる。
Value Objectでドメインの「形」を体現する。

---

## 特性7：局所的なシンメトリー（Local Symmetries）

**意味**: 全体的な対称性より、局所的な対称性の変化や部分的な対称性の重なりが
生命の質を高める。

**コードでの現れ**:
- 類似した操作が一貫した命名・構造で表現されている
- CRUDやコマンド群が対称的なインターフェースを持つ
- 同じパターンが局所的に繰り返されている

**弱さの兆候**:
- 同じ種類の操作がまったく異なるインターフェースを持つ
- 命名規則が場所によってバラバラ
- 一部だけ例外的な実装がある（その例外に理由がない）

**変容の方向**:
局所的な命名・構造の対称性を整える。
ただし、機械的な全体統一は避ける（特性11「粗っぽさ」が失われる）。

---

## 特性8：深い相互結合（Deep Interlock and Ambiguity / 両義性）

**意味**: 境界に沿って深い相互結合が生まれることがある。
分離された領域の間に非常に強いつながりを作り、
一方の面と他方の面を明確に区別することを難しくする。

**コードでの現れ**:
- モジュール間の境界が「厚い」（薄いインターフェースではなく、
  双方向のやり取りが豊か）
- ドメインイベントが2つのモジュールを有機的に結合している
- 関数合成が自然に連鎖する

**弱さの兆候**:
- 人工的な疎結合（ドメインの本来の関係を隠している）
- イベントが一方向にしか流れていない
- コンテキスト間の連携が難しい

**変容の方向**:
ドメインが本来持つ深い関係を構造に正直に反映させる。
過度な抽象化によってドメインの関係が見えなくなっていないか確認する。

---

## 特性9：コントラスト（Contrast / 違い）

**意味**: 対照的なもの同士のコントラストが新しいものを生み出す。
各センターは周囲との違いによって強化される。

**コードでの現れ**:
- 正常系と異常系の処理が明確に分離されている
- コマンドとクエリ、副作用あり/なし が明確に区別されている
- 成功パスと失敗パスが対比的に表現されている

**弱さの兆候**:
- 正常系と例外処理が混在している
- 副作用のある処理とない処理が区別されていない
- 条件分岐が複雑で、どのケースが「正常」かわからない

**変容の方向**:
副作用を分離する。Early returnでコントラストを明確化する。
Result型/Option型でコントラストを型に反映する。

---

## 特性10：グラデーション（Gradients / 段階的変容）

**意味**: 質の突然の変化ではなく、なだらかな変異がある領域が存在する。
センター群は大きさ・距離・強度において非常になだらかな変異を示す。

**コードでの現れ**:
- 抽象化の深さが重要性に応じてなだらかに変化する
- 設定値やパラメータが段階的に調整できる
- 機能のオン/オフではなく、グラデーションで制御できる部分がある

**弱さの兆候**:
- 二値的なフラグだらけ（オン/オフしかない）
- 抽象化の深さが突然変わる（薄い層と厚い層が交互）
- 設定が硬直的で段階的な調整ができない

**変容の方向**:
ストラテジーパターンやポリシーオブジェクトで段階的な変化を表現する。
設定の柔軟性を高めながら、デフォルトを「最もよい形」にする。

---

## 特性11：粗っぽさ（Roughness / 個別性）

**意味**: すべての生物は個性的。ある種のさりげなさや形態学的な粗っぽさを持ち、
それは理想に相当する形からは程遠い。自然界には2つとして同じものが存在しない。

**コードでの現れ**:
- 重要なドメインロジックは丁寧に、周辺は簡潔に（均一でない）
- 実装がそのドメインの個別性を正直に反映している
- 「完璧に整えること」を目的にしていない

**弱さの兆候**:
- すべての箇所を同じレベルで抽象化しようとしている
- パターンの機械的適用（意味のないファクトリ、意味のないアダプター）
- DRY原則の過適用（文脈の異なるものを無理に共通化）

**変容の方向**:
過度な均一化を避ける。
「違うものは違って書く」という判断を意識的に行う。

---

## 特性12：共鳴（Echoes / 類似性）

**意味**: 同じプロセスから生まれた部分どうしは、家族的類似性を持つ。
その形から「関連がある」「仲間である」と見なせる。

**コードでの現れ**:
- 命名規則の一貫性（同じ概念は同じ名前で）
- 同じ問題を同じアプローチで解く一貫性
- テストと実装の構造的対応
- コードの「語り口」が全体を通じて一致している

**弱さの兆候**:
- 同じ概念に複数の名前（User, Account, Member が混在など）
- 場所によってまったく異なる解決アプローチ
- ユビキタス言語がコードに反映されていない

**変容の方向**:
ユビキタス言語を見直し、コード全体に浸透させる。
リファクタリングで命名の一貫性を高める。

---

## 特性13：空（The Void / オープンスペース）

**意味**: もっとも深遠なセンターには真ん中に「空」があり、
底知れぬ媒体のように機能する。生きているシステムはセンターたちと詳細だけでは構成できない。

**コードでの現れ**:
- 関数・クラス間の適切な「余白」（空行、空のモジュール境界）
- 意図的に「何もしない」実装（Null Object, No-op）
- 将来の拡張のための意図的な空白
- 詰め込まれていない設計

**弱さの兆候**:
- すべてのメソッドが何かをしている（Null Objectがない）
- 将来のためのフラグが無数にある（空白を埋めようとしている）
- コードが「詰め込まれた」印象を与える

**変容の方向**:
Null Object パターンを導入する。
過剰な実装を削除し、空白を作る。
「何も書かないこと」を設計として認識する。

---

## 特性14：シンプルさと静謐さ（Simplicity and Inner Calm）

**意味**: 生きているシステムの質はセンターの密度と強度が高まると高まるが、
同時に、不要なものがすべて取り除かれたシンプルさと内なる平穏が重要な特性。

**コードでの現れ**:
- 読んだときに「なるほど」と即座に感じられる構造
- Principle of Least Astonishment が実現されている
- 変更するとき不安を感じない
- 「賢さ」よりも「明確さ」が優先されている

**弱さの兆候**:
- 読んで不安になるコード
- 「賢すぎる」実装（トリッキーな最適化など）
- 理解するのに長い時間が必要
- 変更前に必ず全体を把握しなければならない

**評価方法**:
このコードを初めて見る人が、コアロジックを理解するのに何分かかるか？
5分以内なら良好。30分以上かかるなら改善が必要。

**変容の方向**:
複雑な条件分岐を名前付きの概念に変換する。
「賢い」コードを「明確な」コードに書き直す。

---

## 特性15：不可分性（Not-Separateness / つながっていること）

**意味**: アレグザンダーによると、この特性は他の14の特性と比べて特別で唯一のもの。
他の特性はシステムを美しくできるが、「自分を見て、いかに美しいか」という
自己中心的な性格がある。不可分性を持つものは、深い謙虚さを示す。
「それらは全体に奉仕し、癒す力を持つ」。

**コードでの現れ**:
- すべてのコードが主要なユースケースフローと有機的に繋がっている
- アーキテクチャの哲学がモジュールレベルでも反映されている（フラクタル性）
- 孤立したモジュールがない
- デッドコードがない（繋がりが断たれたコード）
- 部分と全体が同じ言語・同じ哲学で語られている

**弱さの兆候**:
- 到達不可能なコード・デッドコード
- 使われていない抽象層・インターフェース
- モジュールがシステム全体から「孤島化」している
- 部分を見ても全体の哲学が感じられない

**検出**:
```bash
# デッドコード候補（参照されていない定義）
# 言語によって手法は異なる
```

**変容の方向**:
使われていないコードを削除する。
部分の設計が全体の設計哲学と一致しているか確認する。
「全体に奉仕する」という観点でリファクタリングする。

## 15特性の診断チートシート

```
弱いと感じる特性に☑をつけ、変容の優先候補とする

□ 特性1  スケールの段階性  ← 粒度の問題
□ 特性2  力強いセンター    ← 責務の散在・God Object
□ 特性3  境界             ← インターフェースの曖昧さ
□ 特性4  交互反復          ← データ/処理の混在
□ 特性5  正の空間           ← 責務の空白・重複
□ 特性6  よい形            ← ドメインの形が見えない
□ 特性7  局所的シンメトリー  ← 命名・構造の不一致
□ 特性8  深い相互結合       ← 過度な疎結合・薄すぎる境界
□ 特性9  コントラスト       ← 正常系/異常系の混在
□ 特性10 グラデーション     ← 二値的フラグだらけ
□ 特性11 粗っぽさ          ← 過度な均一化・DRY過適用
□ 特性12 共鳴             ← 命名の不一致・語り口のバラバラ
□ 特性13 空               ← 詰め込まれた設計・Null Objectなし
□ 特性14 シンプルさと静謐さ ← 読んで不安・賢すぎる実装
□ 特性15 不可分性          ← デッドコード・孤島化したモジュール
```
