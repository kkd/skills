# プロダクト全体性と機能の構造保存変容

## 根本的な前提

**全体性の「場」は利用者の生活の中にある。**

コードレイヤーでは「コードベースが全体性の場」だった。
プロダクトレイヤーでは違う。プロダクトの全体性は利用者の
生活・文脈・悩みの中に宿っており、プロダクトはその全体性に
「参加している」に過ぎない。

機能追加とは「新しい価値を加える」ことではなく、
**利用者の全体性の中にすでに潜在しているセンターを顕在化させること**だ。

アレグザンダーが盈進学園で実践したことと同じ：
建物が先にあるのではなく、そこで生きる人々の生活の全体性が先にあり、
建物はその全体性に奉仕するように生まれた。

---

## プロダクト全体性のセンター構造

利用者とプロダクトの関係は、以下のセンター階層として捉えられる：

```
スーパーセンター（利用者の生活全体）
  │
  └── 利用者文脈センター（悩み・環境・習慣・関係）
        │
        ├── コアバリューセンター（プロダクトが奉仕する根本的な「仕事」）
        │     │
        │     ├── フィーチャセンター（個別機能）
        │     ├── フィーチャセンター
        │     └── フィーチャセンター
        │
        └── 境界センター（「このプロダクトがすることとしないこと」）
```

| センター | 意味 | 弱さの兆候 |
|---------|------|-----------|
| **利用者文脈センター** | 利用者の生活・悩み・環境の構造全体 | 利用者の「前後」が見えていない |
| **コアバリューセンター** | プロダクトが奉仕する根本的な「仕事」（JTBD） | 「何のためのプロダクトか」が内部で揺れている |
| **フィーチャセンター** | 個別機能。コアバリューを支えるサブセンター | 機能が多いのにコアが弱い |
| **境界センター** | プロダクトがしないことの輪郭 | 何でもできるが何かに優れてもいない |
| **インタラクションセンター** | 利用者とプロダクトが接する瞬間・流れ | 使われているが「体験」が感じられない |

---

## アレグザンダーの5ステップ：プロダクトレイヤー版

### STEP 1: 利用者の全体性を読み取る

変更・追加の前に必ず行う「感じ取り」。
インプットの種類によって読み取り方が異なる。

---

#### インプット A：ユーザーインタビュー・観察記録

インタビュー記録を読むとき、**発言の内容より「前後の文脈」を読め**。

```
## 全体性読み取りシート（インタビューから）

利用者が「その場面」にたどり着くまでに何があったか:
→ [記録から抜き出す]

その場面の後、利用者は何に向かうか:
→ [記録から抜き出す]

利用者がプロダクトなしで「なんとかしてきたこと」は何か:
→ [workaround・代替手段を特定する]

利用者が言葉にしていないが、繰り返し現れる「動き・パターン」は何か:
→ [潜在センター候補]

利用者が「うんざり」した瞬間・「なんとかしてくれれば」という声はどこか:
→ [もっとも弱いセンターの手がかり]
```

**注意**: 利用者の「要望」は潜在センターへの手がかりであって、
そのまま機能要件ではない。「もっと速くしてほしい」は
「遅さが全体性を壊している」というシグナルだ。

---

#### インプット B：ユーザーストーリー・バックログ

既存のバックログを「センターマップ」として再読する。

```bash
# （エージェントの作業）バックログを以下の観点で分類・整理する

カテゴリ1: コアバリューセンターを直接強化するストーリー
カテゴリ2: 境界センターを整えるストーリー
カテゴリ3: インタラクションセンターを強化するストーリー
カテゴリ4: フィーチャセンターの個別改善
カテゴリ5: センターが不明確・コアとの接続が見えないストーリー
```

バックログ全体を見渡して問え：

- カテゴリ5（センター不明確）が多い → コアバリューセンターが弱い兆候
- カテゴリ1が少ない → コアから離れた変容が続いている
- 境界センターが曖昧 → 「何でもできる」プロダクトへの漂流が始まっている

---

#### インプット C：アナリティクス・行動ログ

数字の背後にある「全体性」を読む。数字そのものがセンターではない。

```
## 全体性読み取りシート（ログから）

最も使われているフィーチャ（強いセンター候補）:
→ [上位3つ]

使われていないフィーチャ（弱いセンター・孤島化の兆候）:
→ [特性15「不可分性」が失われている可能性]

利用者が離脱する瞬間・箇所:
→ [全体性の流れが断ち切られている場所]

予想外の使われ方・意図していない経路:
→ [利用者が自力で作った「非公式センター」。潜在センターの可能性]

特定の操作が繰り返される・手数が多い箇所:
→ [workaround。潜在センターが顕在化していない場所]
```

---

### STEP 2: もっとも弱い部分を特定する

> 「構造のもっとも弱い部分を、もっとも感情が感じられない部分として特定する」

プロダクトレイヤーで「感情が感じられない」とは：

| シグナル | 意味する弱さ |
|---------|------------|
| 「使えるけど、好きじゃない」という声 | インタラクションセンターが弱い |
| 機能が増えるほど「何のプロダクトか」が見えなくなる | コアバリューセンターが希薄化 |
| 「これできますか？」という問い合わせが多い | 境界センターが不明確 |
| 利用者が「プロダクトのために」行動を変えている | 利用者文脈への不可分性が失われている |
| ヘビーユーザーとライトユーザーの使い方が乖離する | フィーチャセンターが偏って発達している |

---

### STEP 3: 潜在センターを見出し、次の変容を提案する

> 「3つの異なるレベルで潜在的なセンターを探す」

#### 3レベルの探索

**上位（利用者の生活文脈）方向**:
- 利用者はこのプロダクトを使った「後」、何に向かうのか
- その「後」の文脈と、プロダクトの現在の境界の間に隙間はないか
- 利用者の生活の中で、プロダクトが「奉仕できていない部分」はどこか

**下位（フィーチャ内部）方向**:
- 既存のフィーチャの中に、まだ名前が付いていない「機能のかたまり」はないか
- 利用者が「なんとなく一緒にやる」操作セットの中に、顕在化すべきセンターはないか
- 「設定でできます」で対処している部分の中に、コアとして昇格すべきものはないか

**隣接（文脈・環境）方向**:
- 利用者がプロダクトと「一緒に」使っているツール・習慣・関係者は何か
- その隣接センターとの「間の空間」に、整えるべき接続点はないか
- 利用者の職場・家族・コミュニティの中でプロダクトはどう位置づけられているか

---

#### 人間への提案フォーマット

**エージェントは分析と仮説の提示まで行い、判断は必ず人間に委ねる。**

```
## プロダクト全体性の分析と変容提案

【現在の全体性の状態】
[利用者の生活文脈の中でプロダクトがどう位置しているかを1〜3文で描写]

【もっとも弱いセンター】
センター名: [名前]
弱さの根拠:
  - インタビューから: [具体的な発言・行動パターン]
  - バックログから: [関連するストーリーの傾向]
  - ログから: [数字が示す行動パターン]

【潜在センターの候補】（優先順）

候補1: [センター名]
  発見された場所: [上位 / 下位 / 隣接]
  根拠: [どのインプットから、どのシグナルを読み取ったか]
  変容の方向: [どの15特性を強化するか。以下の対応表を参照]
  スモールステップ案:
    1. [最小の変容。これだけでも検証できる]
    2. [次のステップ]
  構造保存の判定:
    - コアバリューセンターを強化するか: Yes / No / 要確認
    - 境界センターを尊重するか: Yes / No / 要確認
    - 利用者文脈との不可分性: [説明]
  リスク: [どんな全体性を損なう可能性があるか]

候補2: [センター名]
  [同上]

【判断をお願いします】
以下についてご確認ください：
1. この全体性の読み取りは、あなたが感じている実態と一致していますか？
2. 候補1から着手することに同意しますか？
3. 私が見落としている文脈・制約はありますか？
```

---

### STEP 4: 最小変容を実施する

機能追加の構造保存変容は、コードレイヤーと同様に**スモールステップ**で行う。

**プロダクトレイヤーのスモールステップ**:

```
潜在センターの発見
  → 最小の形で顕在化する（プロトタイプ・フィーチャフラグ）
    → 利用者の全体性の中で機能するか検証する
      → 全体性が強まったなら次のステップへ
      → 全体性が弱まったなら元に戻す
```

**ユーザーストーリーの記述フォーマット（構造保存変容版）**:

```
## Story: [タイトル]

潜在センターの記述:
  利用者の文脈: [どんな生活・仕事の文脈にいるか]
  現在の弱さ: [どのセンターが弱く、何が起きているか]
  潜在センター: [顕在化させようとしているものの名前]

従来のユーザーストーリー形式:
  As a [利用者], I want to [機能], so that [価値]

構造保存の判定:
  □ コアバリューセンターを強化する
  □ 利用者文脈との不可分性を保つ
  □ 境界センターを尊重する
  □ スモールステップで実施できる
  □ 検証して元に戻せる

15特性での対応:
  強化する特性: [特性番号と名前]
  根拠: [なぜその特性が弱いと判断したか]

完了の定義（DoD）:
  □ 機能が動作している
  □ 利用者の「前後の文脈」との接続が自然である
  □ コアバリューセンターの密度が維持/向上している
  □ 全体性スナップショットを記録した
```

---

### STEP 5: 変容をテストし、元に戻す判断をする

**成功の感触**（プロダクトレイヤー）:
- 「これが欲しかった」という声が、機能の説明なしに生まれる
- 利用者が機能をプロダクトの「一部として」使う（孤立した道具ではない）
- コアバリューセンターへの参照・言及が増える

**元に戻す判断基準**:
- 機能が「使われているが、コアバリューと結びついていない」
- 「これ何のためにあるの？」という問いが増える
- 新機能が「生命の質の高いコア」より目立ち始める（特性15の劣化）

---

## 15特性のプロダクトレイヤー対応表

| 特性 | プロダクトでの現れ | 弱さの兆候 |
|------|----------------|-----------|
| 1 スケールの段階性 | フィーチャに大中小の粒度がある | すべての機能が同等に扱われている |
| 2 力強いセンター | コアバリューが明確で強い | 「何のプロダクトか」が答えにくい |
| 3 境界 | しないことが明確 | 何でもできるが何かに優れていない |
| 4 交互反復 | 入力と出力、問いと答えのリズム | 一方的なアウトプットのみ |
| 5 正の空間 | フィーチャ間の「間」も意味がある | 機能同士が無関係に並んでいる |
| 6 よい形 | UIが利用者の動きに適応している | 利用者がプロダクトに合わせている |
| 7 局所的シンメトリー | 類似操作に一貫したパターン | 同じことが場所によって違うやり方 |
| 8 深い相互結合 | 機能同士が有機的に連携している | 機能が孤立したツールの集まり |
| 9 コントラスト | できることとできないことが明確 | 何でも「できます」という曖昧さ |
| 10 グラデーション | 初心者〜上級者への段階的適応 | 二値的なオン/オフしかない |
| 11 粗っぽさ | プロダクトの個性・語り口がある | 無個性で交換可能な印象 |
| 12 共鳴 | 全体を通じて同じ「言語」で語られる | 機能ごとに概念・用語がバラバラ |
| 13 空 | 余白・「しない領域」が存在する | 詰め込まれた機能リスト |
| 14 シンプルさと静謐さ | 使い始めて即座に「わかる」感覚 | 使い方の説明が必要 |
| 15 不可分性 | 利用者の生活に自然に溶け込む | 「プロダクトのために」行動を変えている |

---

## 構造破壊のシグナル（機能追加で現れる危険な兆候）

これらが現れたら**即座に立ち止まり、人間に確認を求めよ**：

- 「競合がやっているから追加する」
  → 利用者文脈を無視した外部都合の変容。コアバリューセンターを希薄化させる

- 「ユーザーが要望した」だけで追加する
  → 表面的欲求への対応。潜在センターかどうかを先に問え

- 「あると便利」程度の機能の蓄積
  → 境界センターの形骸化。特性3（境界）と特性13（空）の劣化

- 機能が増えるほど「何のプロダクトか」が薄れる
  → コアバリューセンターの密度低下。特性2（力強いセンター）の劣化

- 「パワーユーザー向け」機能の蓄積
  → 特定センターへの偏りと境界の歪み

- リリース後「使われているが愛されていない」状態
  → 特性15（不可分性）が達成されていない。生活文脈に溶け込んでいない
