# Layer I / Layer W: インパクトと世界の全体性

---

## Layer I: 利用者へのインパクト

### Layer I の前提

Layer I は「使った結果、利用者の文脈・能力・関係性がどう変わるか」を問う。

| | 問い | 時間軸 |
|---|---|---|
| Layer U | 使いやすいか？（操作の瞬間） | 今、この操作 |
| Layer P | 役に立つか？（機能・体験） | 使用セッション |
| **Layer I** | 利用者の文脈・能力・関係性がどう変わるか | 使い続けた先 |

Layer U が高くても Layer I が低い例:
SNSのフィード — スクロールは快適（U高）だが、使うほど集中力・時間・対面関係が侵食される（I低）

**設計意図の詳細 → `DESIGN-NOTES.md`（Layer I の設計意図 セクション）**

---

### Layer I の主要センター

| センター種別 | 問い | 健全な状態 |
|---|---|---|
| 自律センター | 使うほど自分でできることが増えるか | 使い続けると利用者が成長する |
| 依存センター | 使うほどこれなしでは機能できなくなるか | 依存が愛着・信頼から来ている |
| 開放センター | 選択肢が広がるか、狭まるか | 利用者の可能性が広がる |
| 接続センター | 他者との関係を豊かにするか、希薄にするか | コミュニティが強化される |
| 能力センター | 人間の能力を伸ばすか、置き換えるか | 人間が主体であり続けられる |

「依存」が即座に悪いわけではない。
依存が「愛着・信頼」から来るなら健全なセンターかもしれない。
依存が「スイッチングコスト・通知・習慣化強制」から来るなら癌的なセンターだ。

---

### Layer I のセンター検出プロトコル

#### Step 1: エージェントが構造から読める範囲

```
確認する観点:
□ このプロダクトの主な使用パターン（繰り返し使用か、一時利用か）
□ 利用者が「これがないとできない」と感じる機能はどれか
□ 利用者の能力・知識が使用を通じて変化しているか
□ 利用者の他の活動・関係性への影響（ポジティブ・ネガティブ）
□ B×I 張力: 解約防止のために依存が設計されていないか
```

#### Step 2: 人間への問い返し — インパクトの感覚を聞く

> 「このプロダクトを使い続けた利用者について、感覚として聞かせてください。
>
> **① 「使うほど、利用者が自分でできることが増えている気がする」場所はありますか？**
>    成長・開放・強化を感じる部分。
>
> **② 「使うほど、利用者がこれなしでは動けなくなっている気がする」場所はありますか？**
>    それは愛着からですか？それとも囲い込みからですか？
>
> **③ 「このプロダクトが利用者の生活・関係・時間に与えている影響」で、**
>    **気になっていること・後ろめたいことはありますか？**
>
> 言葉にならなくてもかまいません。」

```
受け取り方:
①「自律が増えている」  → 健全な I センター。変容で守るべき構造
②「依存している」     → 愛着起因か囲い込み起因かを確認する
                       囲い込み起因なら B×I の癌的張力として変容候補
③「気になる・後ろめたい」→ I の弱いセンター。Layer W の問いにつながる可能性
```

#### Step 3: I 層センターマップ

```
## I 層センターマップ

自律センター:
  使うほど利用者が成長する部分: [あれば]
  使うほど利用者が依存する部分: [あれば]
  依存の性質: [愛着 / 囲い込み / 不明]

開放センター:
  選択肢が広がる部分: [あれば]
  選択肢が狭まる部分: [あれば]

接続センター:
  関係を豊かにしている部分: [あれば]
  関係を希薄にしている部分: [あれば]

能力センター:
  人間の能力を伸ばしている部分: [あれば]
  人間の能力を代替している部分: [あれば]

潜在センター（育てたら利用者インパクトが高まりそうな部分）:
  [候補と理由]
```

## 15特性 × Layer I 対応表

| 特性 | 利用者インパクトでの現れ | 弱さの兆候 | 変容の方向 |
|------|----------------------|-----------|----------|
| 1 スケールの段階性 | 初心者→中級者→上級者への段階的な関わり方の変化。成長フェーズごとのサポート粒度が違う | 全ての利用者に同じ操作・情報・サポートを提供している。成長段階が設計されていない | 利用者の成長段階に応じてプロダクトの関与密度を変える |
| 2 力強いセンター | 「どの能力・自律を育てるか」が明確。全機能がその能力の成長を中心に整理されている | 「使って何が育つのか」が答えられない。機能ごとに利用者インパクトの方向がバラバラ | 育てようとする能力センターを命名し、機能をその観点で整理する |
| 3 境界 | プロダクトが「代替する能力」と「育てる能力」の境界が意識されている | 全ての作業をプロダクトが代替しようとしている。利用者が考える余地がない | 代替すべき作業と利用者に残すべき作業の境界を設計する |
| 4 交互反復 | 利用者の「挑戦」とプロダクトの「支援」のリズム。難しすぎず易しすぎず交互に展開 | 常に支援が続く（利用者が挑戦しない）か、常に挑戦のみ（支援がない）か | 「やってみて→フィードバックを受けて→また挑戦する」リズムを設計する |
| 5 正の空間 | 利用者が「自分でやる余地」が意図的に確保されている。プロダクトが全部解決しない | プロダクトがあらゆる作業を代替しようとしている。利用者が試す機会がない | 利用者に残す「余白」を意図的に設計する。全部解決しないことを設計判断として認識する |
| 6 よい形 | プロダクトの介入量・タイミングが利用者の能力曲線・成長曲線に適応している | 利用者のスキルに関わらず同じ量の支援を提供している。成長しても変わらない | 利用者の習熟度に応じてプロダクトの「形」が変わるように設計する |
| 7 局所的シンメトリー | 似た成長フェーズには一貫したサポートパターンがある。利用者が「次はどうなるか」を予測できる | 成長フェーズごとにまったく異なるアプローチ。利用者がパターンを学習できない | 成長フェーズに応じた一貫したサポートパターンを設計する |
| 8 深い相互結合 | プロダクトと利用者の能力が相互に影響し合っている。プロダクトが育つほど利用者も育つ関係 | プロダクトが一方的に便利さを提供するだけ。利用者とプロダクトの関係に深みがない | 利用者の行動がプロダクト自体を改善する仕組みを設計する（パーソナライズ・学習・フィードバックループ） |
| 9 コントラスト | 自律を促す機能と依存を生む機能のコントラストが意識されている。どちらを選んでいるかが設計に現れている | 自律的依存（愛着）と囲い込み的依存（スイッチングコスト）が混在・無自覚 | 各機能について「これは自律を促すか、依存を生むか」を問い、意図的にコントラストをつける |
| 10 グラデーション | 依存→自律への移行経路が設計されている。使い続けるほどプロダクトへの依存度が下がる（または深まる愛着に変わる） | 利用者がいつまでも初心者扱い。自律への移行経路がない | 「利用者が自律した後」の関係性を設計する。段階的な「手放し」のプロセスを作る |
| 11 粗っぽさ | 特定の利用者・文脈に向けた「個性的な支援」がある。汎用的なツールでなく、特定の人のための道具になっている | 全ての利用者に同じ支援。「誰にでも使えます」は「誰のためにもなっていない」に等しい | 最も大切にしたい利用者の文脈を特定し、その人のためだけの細部を設計する |
| 12 共鳴 | プロダクト全体を通じて「利用者の成長・自律」というテーマが一貫している | 機能によって「自律支援」と「依存促進」が混在している。一貫したインパクトの方向性がない | 全機能について「この機能は利用者の自律を支援するか」を問い、一貫した方向性を持たせる |
| 13 空 | 利用者が「自分でやる」空間が意図的に残されている。プロダクトが全ての問題を解決しない | プロダクトが全ての問題を解決しようとする。利用者が考える・試す・失敗する余地がない | 「プロダクトが介入しない領域」を意図的に設計する。利用者の試行錯誤の余地を残す |
| 14 シンプルさと静謐さ | プロダクトを使うほど利用者の認知負荷が下がる。使い込むほど「楽になる」感覚 | プロダクトを使うほど管理すべきことが増える。使い込むほど複雑になる | プロダクトが利用者の代わりに複雑さを吸収する設計を意識する。利用者に見せる情報を絞る |
| 15 不可分性 | プロダクトの支援が利用者の能力・自律と一体化している。使うほど「自分の力」として感じられる | 利用者がプロダクトに依存しているが「自分の力」とは感じていない | プロダクトの貢献が利用者の能力として内面化されるように設計する。「プロダクトの力」から「自分の力」への転換を意識する |

## Layer W: 社会×世界の全体性

### Layer W の前提

Alexanderは「建物の美しさ」を個別建築の問題として捉えなかった。
「その建物が都市を生き生きさせるか、それとも死なせるか」まで射程に入れた。

プロダクトに置き換えると：
> 「このプロダクトが広く普及した世界は、より生き生きしているか？」

Layer W は**分析・診断ではなく、問いを持ち続けること**として定義する。
常にすべての変容に適用するのではなく、特定のタイミングで問いを立てる。

**設計意図の詳細 → `DESIGN-NOTES.md`（Layer W の設計意図 セクション）**

---

### Layer W を問うタイミング

以下のタイミングで Layer W の問いを立てる。それ以外では省略して構わない。

1. **大きな設計判断をするとき**（アーキテクチャ変更、ビジネスモデル変更）
2. **「後ろめたさ」の感覚が出てきたとき**（Layer B・I の感覚マッピングで検出）
3. **STEP 5（振り返り）の節目**（D モードのみ）

---

### Layer W の問いかけ

> 「少し立ち止まって、大きな視点で聞かせてください。
>
> **「このプロダクトが10年後に広く使われている世界を想像したとき、どう感じますか？**
>  より生き生きしている感じがしますか？それとも何か後ろめたいですか？」

```
受け取り方:
「生き生きしている」 → W センターは健全。そのまま進む
「後ろめたい」      → W のシグナル。Layer B・I のどこで犠牲が生じているか掘り下げる
「わからない」      → それで構わない。問いを持ち続けることに意味がある
```

---

### 「後ろめたさ」の背後にある W のパターン

「後ろめたさ」を感じたとき、以下のパターンが背後にある可能性がある。

| 表面の感覚 | W レベルでの問い |
|---|---|
| 売れているが後ろめたい | 誰が犠牲になって収益が成り立っているか |
| 便利だが何かがおかしい | コミュニティ・環境・資源への影響は？ |
| ユーザーは満足しているが | 格差・分断・孤立を深めていないか |
| 正しいことをしている気がしない | 世界の全体性を強めているか、弱めているか |

---

### W の問いを変容に反映する方法

Layer W の問いに「後ろめたさ」の答えが来たとき、
直接 W を変容しようとするのではなく、
**B・I の変容に W の観点を反映させる**。

```
「後ろめたさ」のシグナル → Layer B・I のどこに起因するか特定する
                        → B・I の変容候補として道筋に組み込む
                        → 変容後に W の感覚が変わったか問い返す
```

Layer W は「変容の目標」ではなく「変容の羅針盤」として使う。
