# 15特性 × 全レイヤー横断リファレンス

> 各特性が C（コード）・U（UI/UX）・P（プロダクト）・I（利用者インパクト）・B（ビジネスモデル）の
> 5つのレイヤーでどのように現れ、どこが弱くなり、どう変容できるかを横断的に示す。
>
> 詳細な単レイヤー記述:
> - Layer C → `fifteen-properties.md`
> - Layer U → `ui-ux-wholeness.md`
> - Layer P → `product-wholeness.md`
> - Layer I → `impact-wholeness.md`
> - Layer B → `business-model-wholeness.md`

**Layer H 観察**は各特性の弱さが対話に現れるシグナルを記す。診断テーブルではなく観察のヒント。

## 特性1：スケールの段階性（Levels of Scale）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | System→Module→Class→Method の自然な階層。ディレクトリ構造が意味的階層を表現 | 全てが同じ粒度（平坦構造）。1クラスが数千行、または1段の浅い構造 | 巨大クラスを適切なサブセンターに分解。大中小の粒度を揃える |
| **U** | 見出し>本文>キャプションの視覚階層。コンポーネントの大中小の粒度差 | 全ての要素が同じ視覚的重みを持つ。階層が読めない画面 | タイポグラフィとコンポーネントの粒度差を意識的に設計する |
| **P** | コア機能>サブ機能>マイクロインタラクションの粒度差。機能の「大中小」がある | 全機能が同等に扱われている。ランディングページに機能が羅列されている | 機能をコアバリューとの距離で階層整理する |
| **I** | 初心者→中級者→上級者への段階的な関わり方の変化。成長フェーズごとのサポート粒度 | 全ての利用者に同じ操作・情報を提供している。成長段階が設計されていない | 利用者の成長段階に応じてプロダクトの関与密度を変える |
| **B** | 個人→SMB→Enterprise の顧客セグメント階層。価格帯・サービス内容の段階性 | 全顧客が同一プランに押し込まれている。ティアの意味的差異がない | セグメントの価値観の違いを基準に階層を設計する |

**Layer H 観察**: 話が「システム全体」か「実装詳細」に一気に飛ぶ → 中間レベルのセンターが共有されていない。

**横断パターン**: どのレイヤーでも「全部同じ粒度」になるとき、核となる中間センターが欠けている。
根の仮説: コアセンターが不明確なために、中間を設計できていない（特性2の弱さが根因）。

## 特性2：力強いセンター（Strong Centers）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | ドメインの中心概念が明確なクラス/モジュールとして体現。参照の集中点が意味的に重要な箇所と一致 | 「このロジックはどこに書くべきか」が自明でない。God Object として肥大化 | 散在したロジックをドメインセンターに集約。God Object は先に弱いセンターを見つけて分離 |
| **U** | 画面の主目的が一目でわかるフォーカスポイント。迷わず目が向かうCTA | CTAが複数あり、どれが主かわからない。「何をする画面か」が0.5秒でわからない | 主アクションの視覚的重みを上げ、他の要素のコントラストを下げる |
| **P** | コアバリュープロポジションが一文で言える（JTBD の核心）。全機能がコアに向いている | 「何のためのプロダクトか」が社内で揺れている。機能が多いのにコアが弱い | コアを一文で定義し、全機能をその一文との距離で再評価する |
| **I** | 「どの能力・自律を育てるか」が明確。全機能がその能力の成長を中心に整理されている | 「使って何が育つのか」が答えられない。機能ごとに利用者インパクトの方向がバラバラ | 育てようとする能力センターを命名し、機能をその観点で整理する |
| **B** | コアバリュープロポジションが一文で言える。主要収益源が明確で揺れていない | 「何で稼いでいるか」が社内で揺れている。複数の収益仮説が競合している | 主要収益源を一本化し、他を補完として位置づける |

**Layer H 観察**: 「このプロジェクトは何のためか」を聞いたとき答えが人によって違う → コアセンターが対話でも共有されていない。

**横断パターン**: P のコアが曖昧 → C に God Object、U に「何の画面か」不明な画面、B に「何で稼ぐか」の揺れ、I に「何を育てるか」の迷いが連鎖して現れる。
根の仮説: プロダクトの存在理由（JTBD）が組織・コードベース・UIのどこにも体現されていない。

## 特性3：境界（Boundaries）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | モジュール/パッケージのインターフェースが明示的。公開するものと内部に留めるものが明確に分かれている | 暗黙のグローバル状態がある。パッケージをまたぐ直接参照がある | インターフェースを明示化し、内部実装を隠蔽する。パッケージ境界にファサードを置く |
| **U** | カード・モーダル・セクション区切りの明確さ。コンテンツ領域の境界が視覚的に明示されている | コンテンツ領域の境界が曖昧。要素がどのグループに属するか読めない | セクション区切りを強化する。関連するものを境界で囲む |
| **P** | 「しないこと」が明確。スコープクリープを防ぐ判断基準がある | 「何でもできるが何かに優れていない」状態。顧客要求に引っ張られて範囲が拡散 | 「やらないことリスト」を明文化する。新機能提案に対して境界センターで判断する |
| **I** | プロダクトが「代替する能力」と「育てる能力」の境界が意識されている | 全ての作業をプロダクトが代替しようとしている。利用者が考える余地がない | 代替すべき作業と利用者に残すべき作業の境界を設計する |
| **B** | 事業ドメインの境界が明確。「やらないこと」が経営判断の基準になっている | 顧客の要求に引っ張られ事業領域が拡散している。「何の会社か」が外から見えない | 「この会社がすること・しないこと」を一文で書く。新規事業の基準にする |

**Layer H 観察**: 話が次々と隣接テーマに広がり収束しない → センター希薄化のシグナル。横断プロトコル（センター再提案）の発動を検討する。

**横断パターン**: B の境界（やらないこと）が曖昧 → P の機能が拡散 → U の画面が増えて複雑化 → C のモジュール境界が混乱 → I で「何のための自律か」が不明確に。
根の仮説: 「この組織は何をする組織か」が決まっておらず、各レイヤーの境界設計の根拠がない。

## 特性4：交互反復（Alternating Repetition）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | データ定義と操作の交互配置。テストコードと実装コードのリズミカルな対応（TDD）。ドメイン層とインフラ層の明確な交互構造 | データクラスと処理クラスが混在して区別できない。テストの粒度が実装と対応していない | CQS（コマンドとクエリの分離）を適用する。テストと実装を対応させる構造を意識する |
| **U** | 入力→確認、問い→答え、アクション→フィードバックのリズム。操作と応答が交互に展開する | 一方向の情報提示のみ。操作後に何が起きたかわからない | フィードバックアニメーション・ローディング状態・完了状態を整える |
| **P** | 利用者の入力とプロダクトの出力が交互に展開する。問いと答えのリズム | 利用者が入力し続けるが出力がない（または逆）。一方通行のプロダクト | 入力と出力のリズムを機能設計の基準として明示する |
| **I** | 利用者の「挑戦」とプロダクトの「支援」のリズム。難しすぎず易しすぎず交互に展開 | 常に支援が続く（利用者が挑戦しない）か、常に挑戦のみ（支援がない）か | 支援の密度を段階的に下げる設計。「やってみて → フィードバックを受けて → また挑戦する」リズムを設計する |
| **B** | 価値提供と価値回収（支払い）のリズムが自然に噛み合っている | 価値は提供しているのに支払いタイミングが乖離している。先払いか後払いかが曖昧 | 顧客が「価値を感じた瞬間」と支払いのタイミングをできるだけ近づける |

**Layer H 観察**: エージェントが分析を出し続けて人間の感覚フィードバックが来ない → 対話の交互反復が失われている。立ち止まって感覚を聞く。

**横断パターン**: どのレイヤーでも「一方通行」になっているとき、フィードバックループの設計思想が欠けている。
根の仮説: 利用者・顧客との双方向性よりも「提供」側の論理が優先されている。

## 特性5：正の空間（Positive Space）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | クラス間の「隙間」（通信プロトコル、インターフェース）が意味を持つ。責務の重複がなく、かつカバーされない責務もない | モジュール間の通信が場当たり的。「どのクラスの責任か」が判断できない空白地帯がある | 責務の境界を見直し、センター間の「間の空間」を整える |
| **U** | ホワイトスペースが意味を持つ余白。要素間の空白が「関係性」を表現している | 余白がなく詰め込まれた印象。空白が「埋め忘れ」に見える | 余白を「関係性の表現」として意図的に設計する。詰め込みを減らす |
| **P** | フィーチャ間の「間の空間」（オンボーディング、ヘルプ、更新フロー）が意味を持つ | 機能同士が無関係に並んでいる。「次に何をすべきか」の流れがない | フィーチャ間の接続点を設計する。利用者の「次の一手」が自然に示されるように |
| **I** | 利用者が「自分でやる余地」が意図的に確保されている。プロダクトが全部解決しない | プロダクトがあらゆる作業を代替しようとしている。利用者が考える・試す機会がない | 利用者に残す「余白」を意図的に設計する。全部解決しないことを設計判断として認識する |
| **B** | CS・オンボーディング・更新プロセスが顧客とプロダクトの「間の空間」として機能している | 「売ったら終わり」。更新・継続フローが設計されていない | 顧客との「継続的な接触点」を設計する。契約後の価値提供プロセスを整える |

**Layer H 観察**: 沈黙や「少し考えます」が来たとき、すぐに次の提案で埋めようとする → 正の空間を潰している。沈黙は潜在センターが浮上しているサインかもしれない。

**横断パターン**: センター自体の設計に注力するあまり、センター間の「間」が設計されない。
根の仮説: 「間の空間」が価値を持つという認識がなく、常に何かを「埋める」方向に動いている。

## 特性6：よい形（Good Shape）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | 関数/クラスがそのドメインの「形」を自然に反映。データ構造がドメインの力学（制約・不変条件）に適応 | 汎用的すぎてドメインの形が見えない。データ構造がDBスキーマやAPIの都合を反映しすぎている | ドメインの力学（制約・不変条件）をコードの形に反映させる。Value Object でドメインの「形」を体現する |
| **U** | ボタン・フォーム要素が操作の「形」を自然に表現。ユーザーが触れば「何をするもの」かわかる | ボタンがリンクに見える。入力欄の目的がわかりにくい。操作とUIの形が一致していない | UIの「形」をそれが担う操作の性質から設計する。Affordance を意識する |
| **P** | プロダクトの形（機能構成・フロー）が利用者の文脈・動き方に適応している | 利用者がプロダクトに合わせて行動を変えている。機能があるのに使われない | 利用者の実際の動き方（ワークフロー）を観察し、それに沿った機能構成に変える |
| **I** | プロダクトの介入量・タイミングが利用者の能力曲線・成長曲線に適応している | 利用者のスキルに関わらず同じ量の支援を提供している。成長しても変わらない | 利用者の習熟度に応じてプロダクトの「形」が変わるように設計する |
| **B** | 収益モデルが市場の力学（顧客の予算サイクル・調達プロセス）に適応している | SaaSなのに顧客が年次予算で動けない。課金単位が顧客の意思決定単位とずれている | 顧客の調達プロセス・予算管理の「形」に収益モデルを合わせる |

**Layer H 観察**: 「それはちょっと違う」という言い直しが続く → 提案の形が問題の形に合っていない。もう一度「問題の形」を一緒に確認する。

**横断パターン**: B のビジネスモデルの「形」が市場に合っていない → P の機能に不自然な制約が生まれる → U の操作が不自然に → C の設計が歪む。
根の仮説: 外部の力学（市場・ユーザー文脈）より内部の都合（技術・組織）が優先されている。

## 特性7：局所的シンメトリー（Local Symmetries）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | 類似した操作が一貫した命名・構造で表現されている。CRUDやコマンド群が対称的なインターフェースを持つ | 同じ種類の操作がまったく異なるインターフェースを持つ。命名規則が場所によってバラバラ | 局所的な命名・構造の対称性を整える。ただし機械的な全体統一は避ける（特性11が失われる） |
| **U** | グリッド整合、アイコンの一貫したスタイル。並列要素が対称的に配置されている | 要素が整列していない。アイコンの雰囲気がバラバラ。同じ種類の情報の見せ方が画面ごとに違う | デザインシステムで局所的パターンを統一する。ただし全画面の一律統一は避ける |
| **P** | 類似機能に一貫したパターンと言語がある。同じことが同じやり方でできる | 同じ操作が場所によって違うやり方。機能ごとに異なる概念体系 | 似た機能のインタラクションパターンを統一する。用語の一貫性を整える |
| **I** | 似た成長フェーズには一貫したサポートパターンがある。利用者が「次はどうなるか」を予測できる | 成長フェーズごとにまったく異なるアプローチ。利用者がパターンを学習できない | 成長フェーズに応じた一貫したサポートパターンを設計する |
| **B** | 類似セグメントへのアプローチが一貫している。同じ価値提案が同じ言語で語られる | 営業担当ごとに価値の説明が違う。同じ顧客層に別々のメッセージを送っている | 顧客セグメントごとの価値メッセージを統一する。営業・マーケ・CSで共通言語を作る |

**Layer H 観察**: 対話の中で同じ概念に違う言葉を使い始める → 共有言語が失われているシグナル。ユビキタス言語の確認を促す。

**横断パターン**: B で異なる顧客に別々の言語を使う → P で機能の概念が分裂 → U でデザインパターンが揺れる → C でユビキタス言語が失われる。
根の仮説: 共通の語彙・価値観の共有が組織レベルで維持されていない。成長の中で再統合が行われていない。

## 特性8：深い相互結合（Deep Interlock）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | モジュール間の境界が「厚い」（薄いインターフェースではなく双方向のやり取りが豊か）。ドメインイベントが2つのモジュールを有機的に結合 | 人工的な疎結合（ドメインの本来の関係を隠している）。イベントが一方向にしか流れない | ドメインが本来持つ深い関係を構造に正直に反映させる。過度な抽象化でドメインの関係が見えなくなっていないか確認 |
| **U** | ホバー・フォーカス・トランジションが操作意図と有機的に結びつく。UIの動きが操作の文脈を反映している | アニメーションが飾りで操作感覚と無関係。操作後の状態変化が文脈と連動していない | インタラクションパターンを「操作の文脈」から設計する。フィードバックが操作の意味を伝えるように |
| **P** | 機能同士が有機的に連携している。ある機能を使うと次の機能が自然に活きる | 機能が孤立したツールの集まり。機能間に相乗効果がない | 機能間の「つなぎ目」を設計する。機能を単体でなく文脈の流れの中に位置づける |
| **I** | プロダクトと利用者の能力が相互に影響し合っている。プロダクトが育つほど利用者も育つ関係 | プロダクトが一方的に便利さを提供するだけ。利用者とプロダクトの関係に深みがない | 利用者の行動がプロダクト自体を改善する仕組みを設計する（パーソナライズ・学習・フィードバックループ） |
| **B** | プロダクトが顧客のワークフローに深く統合されている（正のスイッチングコスト）。顧客の成長とプロダクトの成長が連動 | 顧客が「便利」とは言うが代替ツールへの乗り換えを恐れていない。統合が浅い | 顧客のワークフロー・データ・習慣とプロダクトを深く統合する。乗り換えコストではなく「一緒に育つ」関係を設計する |

**Layer H 観察**: 話の接続が突然切れる・話題が飛ぶ → 暗黙の前提が共有されていない層がある。「今の話と前の話はどうつながっていますか？」と問う。

**横断パターン**: B で顧客統合が浅い → P で機能が孤立したツール群になる → U でインタラクションが一方的 → C でモジュールが疎結合すぎる。
根の仮説: 「関係性の設計」より「機能・モジュールの設計」が優先されている。

## 特性9：コントラスト（Contrast）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | 正常系と異常系の処理が明確に分離。コマンドとクエリ、副作用あり/なしが明確に区別 | 正常系と例外処理が混在。副作用のある処理とない処理が区別されていない | 副作用を分離する。Early return でコントラストを明確化。Result型/Option型でコントラストを型に反映 |
| **U** | プライマリ/セカンダリアクションの視覚的差異が明確。文字の可読性。強調と背景の差 | 全てのアクションが同じ視覚的重さを持つ。文字が背景に溶け込む | 主アクション・副アクション・テキストリンクの視覚的重みを明確に差別化する |
| **P** | できることとできないことが明確。コア機能と周辺機能の差がはっきりしている | 「何でもできます」という曖昧さ。コアと周辺の区別がない | コア機能を明確に定義し、周辺機能との差をプロダクト全体で一貫させる |
| **I** | 自律を促す機能と依存を生む機能のコントラストが意識されている。どちらを選んでいるかが設計に現れている | 自律的依存（愛着）と囲い込み的依存（スイッチングコスト）が混在・無自覚 | 各機能について「これは自律を促すか、依存を生むか」を問い、意図的にコントラストをつける |
| **B** | Free/Paid・Basic/Premiumの差異が明確で価値が伝わる。なぜ上位プランを選ぶかが自明 | 全プランの機能差が曖昧。なぜ上位プランを買うのかわからない | 各ティアの「核心的な差異」を一つに絞り込んで明示する |

**Layer H 観察**: 「どちらも大事です」という答えが続く → コントラストを避けている。優先順位の明示を促す。「もし一つだけ選ぶとしたら、どちらですか？」

**横断パターン**: B で価値の差が不明確 → P でコア機能と周辺機能の差が曖昧 → U でプライマリ/セカンダリの区別がなくなる → C で正常系/異常系が混在。
根の仮説: 優先順位をつけることへの回避。「全て大事」は「何も大事でない」と同義。

**Layer H 観察**: 「全部一度にやりましょう」という方針になる → グラデーションなく一気に進もうとしている。「最初の一歩は何ですか？」と問い直す。

**横断パターン**: B でグラデーション（段階的コミット）がない → P でオンボーディングが急進 → U で初心者が操作に迷う → C で設定が二値的フラグだらけ → I で依存→自律の移行経路がない。
根の仮説: 「全か無か」の思考。中間状態を設計することへの不慣れ。

**横断パターン**: B に個性がない → P に個性がない → U に個性がない → C が汎用的すぎる → I で特定の人への特別な支援がない。全層が「無難」に収束する。
根の仮説: リスクを避けるために「無難」を選んでいる。個性を持つことへの組織的な回避がある。

**横断パターン**: B のブランドメッセージが分裂 → P の機能間で概念が揺れる → U でデザインシステムが維持されない → C でユビキタス言語が失われる → I で「育てようとしているもの」が一貫しない。
根の仮説: 組織の成長・チームの分散に合わせて語彙・価値観の再統合が行われていない。

## 特性13：空（The Void）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | 関数・クラス間の適切な「余白」（空行、空のモジュール境界）。意図的に「何もしない」実装（Null Object, No-op）。将来の拡張のための意図的な空白 | すべてのメソッドが何かをしている。将来のためのフラグが無数にある。コードが「詰め込まれた」印象 | Null Object パターンを導入する。過剰な実装を削除し空白を作る。「何も書かないこと」を設計として認識する |
| **U** | エンプティステートの設計。意図的な余白。「何もしない」状態が丁寧に設計されている | エラー時・空状態が設計されていない。コンテンツがないとき何も表示されない | エンプティステート・エラー状態・ローディング状態を設計の一部として扱う |
| **P** | 余白・「しない領域」が存在する。機能リストに「意図的な空白」がある | 詰め込まれた機能リスト。「あると便利」機能の蓄積。空白を許せない開発サイクル | 「次のリリースには入れない」リストを意識的に作る。空白を守ることを設計判断として認識する |
| **I** | 利用者が「自分でやる」空間が意図的に残されている。プロダクトが全ての問題を解決しない | プロダクトが全ての問題を解決しようとする。利用者が考える・試す・失敗する余地がない | 「プロダクトが介入しない領域」を意図的に設計する。利用者の試行錯誤の余地を残す |
| **B** | 事業の「余白」がある。次の収益源・新セグメントへの成長余地が残っている | 現収益源に過最適化されて次の手が見えない。または手をつけていない領域が多すぎる | 事業の「コア」と「余白」を区別する。今はやらないが将来の可能性として意識的に空けておく領域を定義する |

**Layer H 観察**: 全ての問いに即答しようとする → 「空」を恐れている。「わからない」「保留」が許されていない対話になっている。沈黙や「少し考えます」を歓迎する。

**横断パターン**: 過密（詰め込み）がどのレイヤーにも連鎖する。B が過密（収益源が多すぎる）→ P が機能過多 → U が情報過多 → C が詰め込まれた設計 → I で利用者の自分でやる余地がない。
根の仮説: 「何かをしないこと」を設計として認識していない。空白は失敗、充填は成功という思考。

## 特性14：シンプルさと静謐さ（Simplicity and Inner Calm）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |
|---------|------|-----------|----------|
| **C** | 読んだときに「なるほど」と即座に感じられる構造。Principle of Least Astonishment。変更するとき不安を感じない | 読んで不安になるコード。「賢すぎる」実装。理解するのに長い時間が必要 | 複雑な条件分岐を名前付きの概念に変換する。「賢い」コードを「明確な」コードに書き直す |
| **U** | 使い始めて即座に「わかる」感覚。認知負荷の最小化。操作手順の説明が不要 | 使い方の説明が必要。チュートリアルなしでは使えない。選択肢が多すぎて迷う | 表示情報を削減する。「最も重要なことだけ」を優先する。説明が不要になるまでUIを磨く |
| **P** | 使い始めて即座にプロダクトの目的が「わかる」。機能の説明なしに価値が伝わる | 機能の説明が必要。「何ができるプロダクトか」を言うのに時間がかかる | プロダクトの核心を一つに絞る。機能を削ることで静謐さを取り戻す |
| **I** | プロダクトを使うほど利用者の認知負荷が下がる。使い込むほど「楽になる」感覚 | プロダクトを使うほど管理すべきことが増える。使い込むほど複雑になる | プロダクトが利用者の代わりに複雑さを吸収する設計を意識する。利用者に見せる情報を絞る |
| **B** | 料金体系・契約条件が読んですぐわかる。意思決定の認知負荷が低い | 複雑な価格表。「どのプランが自分に合うか」を顧客が判断できない。例外条件が多い | 料金体系をシンプルにする。選択肢を3つ以内に絞る。例外をなくす |

**Layer H 観察**: 説明が長くなる・言い直しが増える → シンプルさが失われているか、まだ核心にたどり着いていない。「一言で言うと何ですか？」と問う。

**横断パターン**: 複雑さがどのレイヤーにも連鎖する。B の価格体系が複雑 → P のオンボーディングが長い → U の操作手順の説明が必要 → C が読んで不安 → I で利用者の認知負荷が高い。
根の仮説: シンプルさは「削減」ではなく「整理」であるという理解が欠けている。不要なものを取り除く勇気がない。

## 特性15：不可分性（Not-Separateness）

| レイヤー | 現れ | 弱さの兆候 | 変容の方向 |

**Layer H 観察**: 対話の結論が次のアクションに自然につながらない → 分析と実践が分離している。「この話し合いで、次に何をしますか？」と問う。

**横断パターン**: どのレイヤーでも「全体から切り離されている」感があるとき、コアバリューが全体に浸透していない。C でデッドコード、U で使われないUI、P でコアと結びつかない機能、I で利用者の生活から浮いている、B で顧客ワークフローから孤立している。
根の仮説: 「誰の何に奉仕するか」が問われないまま各レイヤーが構築されてきた。
