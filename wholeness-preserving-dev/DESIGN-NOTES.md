# Wholeness-Preserving Dev — 設計ノート

このドキュメントは SKILL の設計判断の背景・意図・議論を記録する。
SKILL 本体には含まれないが、なぜそのような構造になっているかを理解するための基盤となる。

---

## レイヤー構造の設計思想

### なぜレイヤーを分けるのか

Alexanderは全体性を「センターが互いを支え合う構造」として捉えた。
コードだけ、UXだけ、ビジネスだけを見ていると、個々のセンターは改善されても
**全体の生命感は失われる**ことがある。

レイヤーを分けるのは「分離して考える」ためではなく、
「それぞれの全体性がどう張力を持ち合っているか」を見るためだ。

### レイヤーの積み上がり

```
Layer W: 社会×世界の全体性（プロダクトが世界を生き生きさせているか）
Layer B: ビジネスモデル×顧客（収益・持続性・Customer との関係）
Layer I: 利用者へのインパクト（利用者のコンテキスト・自律・依存の変化）
Layer P: プロダクト×利用者（機能・体験・User との関係）
Layer U: インターフェース×人（操作感覚・認知・UI全体性）
Layer C: コード全体性（技術構造・15特性・センター）
─────────────────────────────────────
Layer H: 人間×エージェントの関係（すべての層の対話が行われる場）
```

Layer H はレイヤーの「外側」にある。他のすべてのレイヤーが H の中で対話される。

## Layer B（ビジネスモデル×顧客）の設計意図

### なぜ Layer P から独立させるのか

Layer P は「利用者（User）が役立てているか」を問う。
Layer B は「顧客（Customer）がコストを払う価値があるか」を問う。

B2C では User = Customer であることが多いが、
B2B SaaS や BtoB2C では **利用者と費用を出す顧客は同一ではない**。

- 利用者（User）: 実際に操作する人。Layer U・P・I の主体。
- 顧客（Customer）: コストを判断する人（購買担当、経営層、親企業など）。

この乖離を Layer P だけで扱うと、「利用者に良いもの」が「顧客が買うもの」と
ずれていることに気づきにくくなる。

### Layer B は「張力センター」である

Layer B の設計上の最重要原則：

> **Layer B は他のセンターを強めるときに生命的になり、
> 他のセンターを弱めるときに癌的になる。**

| Layer B の状態 | 他レイヤーへの影響 | 全体性 |
|---|---|---|
| 健全（生命的） | B の明確さが C・U・P・I の焦点を強める | 生きている |
| 強すぎ（癌的） | 収益圧力が C・U・P・I を侵食し弱める | 死んでいく |
| 弱すぎ（空洞） | 収益のセンターが存在せず全体が持続しない | 生命力を失う |

### B×他レイヤーの主要な張力パターン

**B×C（技術負債の蓄積）**
- 健全: 収益モデルが明確 → 技術投資の優先順位が定まる → コードのセンターが強まる
- 癌的: 「とにかく出す」圧力 → ショートカット → 技術負債 → 将来の収益を食いつぶす
- 空洞: 収益設計なし → 持続投資なし → コードが朽ちる

**B×U（ダークパターンの誘惑）**
- 健全: 価値に共鳴した顧客が自然に購買 → UXと収益が同方向を向く
- 癌的: 転換率最大化 → ダークパターン・誘導設計 → 利用者の操作感覚を損なう
- 空洞: 課金動線が弱い → UXは良いが誰も払わない

**B×P（利用者≠顧客の乖離）**
- 健全: Customerが払う価値 ≒ Userが得る価値（方向が揃っている）
- 癌的: Customerの要求がUserのニーズを上書きする（B2B SaaSでよく起きる）
- 空洞: Userは喜ぶがCustomerの購買理由にならない

**B×I（自律 vs 依存の設計）**
これが B×I の最も根本的な張力：
- 「利用者が自律するほど解約される」という恐れが B×I の最大の緊張だ
- 癌的 B: 依存を設計することで解約率を下げる（ロックイン・通知・習慣化強制）
- 健全 B: 自律させても払い続けたくなる構造を設計する（信頼・成長・コミュニティ）

構造保存変容の問い：「**この変容は、利用者を自律させながら、払い続ける価値も強めるか？**」

## Layer I（利用者へのインパクト）の設計意図

### Layer U・P との違い

| | 問い | 時間軸 |
|---|---|---|
| Layer U | 使いやすいか？（操作の瞬間） | 今、この操作 |
| Layer P | 役に立つか？（機能・体験） | 使用セッション |
| **Layer I** | 利用者の文脈・能力・関係性がどう変わるか | 使い続けた先 |

Layer I は「使った結果として何が変わるか」を問う。

### 自律 ↔ 依存の軸

Layer I の中心的な対立軸：

- **自律**: このプロダクトを使うほど、利用者は自分でできることが増える
- **依存**: このプロダクトを使うほど、利用者はこれなしでは機能できなくなる

どちらが良い・悪いという単純な話ではない。
「依存」が「愛着」「信頼」から来るなら健全かもしれない。
「依存」が「スイッチングコストによる囲い込み」から来るなら癌的だ。

他の軸：
- 開放 ↔ 囲い込み（選択肢が広がるか、狭まるか）
- 強化 ↔ 代替（人間の能力を伸ばすか、置き換えるか）
- 接続 ↔ 孤立（他者との関係を豊かにするか、希薄にするか）

### Layer U が高くても Layer I が低い例

SNSのフィード：
- Layer U 高: スクロールが快適、コンテンツが次々出る
- Layer I 低: 使うほど集中力・時間・対面関係が侵食される

この乖離を Layer U だけ見ていると発見できない。

## Layer W（社会×世界の全体性）の設計意図

### なぜ含めるのか

Alexanderは「建物の美しさ」を個別建築の問題として捉えなかった。
「その建物が都市を生き生きさせるか、それとも死なせるか」まで射程に入れた。

プロダクトに置き換えると：
「このプロダクトが広く普及した世界は、より生き生きしているか？」

これは測定が難しく、感覚でしか掴めない。
だから Layer W は**分析プロトコルではなく、問いを持ち続けること**として定義する。

### 「後ろめたさ」はシグナルである

「後ろめたさ」は Layer W が弱いサインかもしれない。

Layer B の感覚マッピングで問う「後ろめたさ」（売れているが何か後ろめたい）の背後には、
多くの場合 Layer W の問題がある：

- 利用者個人は便利だが、コミュニティが失われている
- 利用者個人は幸せだが、環境や社会的資源を消費している
- 利用者個人は満足だが、格差・分断・孤立を深めている

「後ろめたさ」を感じたとき、その感覚を Layer W の観点から掘り下げることで、
より深い変容の方向性が見えることがある。

### Layer W は「問いを持つこと」として扱う

Layer W を常にすべての変容に適用するのは現実的ではない。
ただし、以下のタイミングで問いを立てることに価値がある：

1. 大きな設計判断をするとき
2. 「後ろめたさ」の感覚が出てきたとき
3. STEP 5（振り返り）の節目

問いの形：
> 「このプロダクトが10年後に広く使われている世界を想像したとき、どう感じますか？
>  より生き生きしている感じがしますか？それとも何か後ろめたいですか？」

答えが「生き生き」ならそのまま進む。
答えが「後ろめたい」なら、その感覚を Layer B・I に反映させる変容を検討する。
答えが「わからない」なら、それで構わない。問いを持ち続けることに意味がある。

## 7レイヤーの全体構造

```
Layer W: 社会×世界の全体性
  ↕ 張力（プロダクトが世界を侵食しているか、生き生きさせているか）
Layer B: ビジネスモデル×顧客
  ↕ 張力（収益が自律を支えるか、依存を強制するか）
Layer I: 利用者へのインパクト
  ↕ 張力（体験が利用者を強化するか、弱体化させるか）
Layer P: プロダクト×利用者
  ↕ 張力（機能が操作と一致しているか）
Layer U: インターフェース×人
  ↕ 張力（操作感覚がコードと一致しているか）
Layer C: コード全体性
```

すべての張力が Layer H（人間×エージェントの対話）の中で可視化され、
構造保存変容の道筋として共同生成される。

## 設計上の注意点

### Layer W は軽量に保つ

Layer W を重くすると、すべての開発判断に哲学的問答が必要になり、
スキルの実用性が失われる。

Layer W は「感覚チェック」として最軽量に置く。分析・診断は Layer B・I が担う。

### Layer B の「強すぎ」を最も警戒する

開発の現場で最も起きやすい全体性の崩壊は、
「ビジネス要求が技術・UX・利用者インパクトを圧倒する」パターンだ。

エージェントは Layer B の変容提案に際して、
B×C・B×U・B×P・B×I の張力を**必ず**可視化する。
「ビジネス的に正しい」だけを根拠にした変容は採用しない。

### User ≠ Customer の乖離は明示する

B2B コンテキストでは、スキルを起動した時点で
「User と Customer は同じ人ですか？」を確認することを推奨する。
乖離がある場合、Layer P（User視点）と Layer B（Customer視点）を
意識的に分けてセンターを読む。
