# センター検出・分析プロトコル

## センターとは

システム・プロダクト・インターフェース・会話の中で「他の部分がそこを中心に組織されている」箇所。
良い構造も悪い構造もセンターを持つ。4つのレイヤーそれぞれにセンターが存在する。

センターには3つの階層がある：
- **スーパーセンター**（上位）: そのセンターを使う・含む上位の文脈
- **サブセンター**（下位）: そのセンターを構成する内部要素
- **サイドセンター**（隣接）: 同一レベルで隣り合う関連構造

**センターの種類**：
- **顕在センター**: すでに明示されている構造（クラス・モジュール・関数）
- **潜在センター**: まだ明示されていないが、少しの変容で生み出せるセンター
- **隠れセンター**: 意識されていないが実際に機能している構造

## Layer C: コードのセンター検出プロトコル（4ステップ）

### Step 1: 定量的分析

```bash
# ── 参照集中点（センター候補）
grep -r "import\|require\|from\|use " . \
  --include="*.rb" \
  | grep -v "spec\|test" \
  | sed 's/.*require //' | sed "s/.*from '//; s/'.*//" \
  | sort | uniq -c | sort -rn | head -20

# ── 変更頻度ホットスポット（git履歴）
git log --name-only --format="" -- "*.rb" \
  | grep -v "^$" | sort | uniq -c | sort -rn | head -20

# ── ファイルサイズ（責務集中の可能性）
find . -name "*.rb" | xargs wc -l 2>/dev/null | sort -rn | head -10

# ── 循環的複雑度が高いファイル（生命の質が低い可能性）
# 言語ごとのツールを使用（例: Ruby なら flog, Pythonなら radon）
```

### Step 2: 定性的分析（問いのリスト）

以下の問いに答えながら、センターを感じ取る：

1. **ドメインエキスパートが「これ」と指す概念はどこにあるか？**
   → ドメインセンターの特定

2. **新しい機能を追加するとき、必ず通過する箇所はどこか？**
   → フローセンターの特定

3. **バグが起きたとき、最初に調べる場所はどこか？**
   → 最も責任が集中するセンターの特定

4. **「このクラス/モジュールなしでは動かない」と感じるものはどれか？**
   → 構造センターの特定

5. **「変更が怖い」と感じる箇所はどこか？**
   → 生命の質が低い（弱い）センターの特定

### Step 3: センターマップの作成

```
## センターマップ

[対象システム/機能]

スーパーセンター（上位）:
  - [センター名]: [役割]

対象センター（現在地）:
  - [センター名]: [役割と強さ：強/中/弱]

サブセンター（下位）:
  - [センター名]: [役割]
  - [センター名]: [役割]

サイドセンター（隣接）:
  - [センター名]: [関係性]
  - [センター名]: [関係性]

潜在センター（まだ明示されていない）:
  - [センター候補名]: [なぜ潜在しているか]
```

### Step 4: センターの健全性評価

| センター | 強さ | 弱さの原因 | 対応する特性 |
|----------|------|-----------|-------------|
| [名前] | 強/中/弱 | [原因] | [特性番号] |

## 「次に着手すべきセンター」の見出し方

アレグザンダーのステップ3に基づく実践的プロトコル。

### アプローチ1: もっとも弱い感情から見つける

「感情がもっとも感じられない部分」を探す。具体的には：

- コードを読んで**最も「うんざり」する部分**はどこか
- **変更を後回しにしてきた**箇所はどこか
- 「**誰かが何とかしてくれれば**」と思ってきた箇所はどこか

これらは「生命の質が最も低いセンター」であり、
変容のエネルギーが最も必要とされる場所だ。

### アプローチ2: 3レベルでの潜在センター探索

現在着目しているセンターの3方向を探索する：

**上位（スーパー）方向**:
- このセンターを使う側のコードは、何を期待しているか？
- その期待に応えるために、今は存在していない「何か」は何か？

**下位（サブ）方向**:
- このセンターの内部に、まだ名前が付いていない概念はないか？
- 似たようなコードが複数あるとき、それらが共有する本質は何か？

**隣接（サイド）方向**:
- 隣接するセンターとの間の「空間」（インターフェース）は整っているか？
- 隣のセンターが持っている特性で、このセンターが欠いているものは何か？

### アプローチ3: 15特性の欠如から見つける

`fifteen-properties.md` の診断チートシートを使い、
最も多く「弱い」と感じた特性を確認する。

その特性が最も強く欠けているセンターを、次の着手点とする。

## センターの優先度マトリクス

```
               影響範囲
               大          小
        ┌──────────────┬──────────────┐
生命  弱 │   最優先      │   次優先      │
の    　 │ （着手せよ）  │ （計画に入れる） │
質  　　 ├──────────────┼──────────────┤
      強 │   中優先      │   低優先      │
        │ （機会があれば）│ （維持でよい）  │
        └──────────────┴──────────────┘
```

**最優先**（生命の質が弱く、影響範囲が大きい）:
→ 今すぐ変容を計画し、人間に提案せよ

**次優先**（生命の質が弱いが、影響範囲が小さい）:
→ 次のスプリント/イテレーションに含める

**中優先**（生命の質は強いが、影響範囲が大きい）:
→ 今後のアーキテクチャ進化の文脈で検討

**低優先**（生命の質が強く、影響範囲も小さい）:
→ 現状維持。過度な変容は避ける

## Layer P: プロダクト×利用者のセンター検出プロトコル

Layer P のセンター検出は2段階で行う。
まずエージェントがバックログ・Issueを構造分析し、次に人間のモヤモヤ感覚で補正する。

### Step 1: バックログ・Issue の構造分析

```bash
# GitHub Issues の一覧（ラベル・マイルストーン付き）
gh issue list --limit 100 --json number,title,labels,milestone,createdAt,updatedAt

# 長期間オープンのまま放置されているIssue（誰も触れないセンター候補）
gh issue list --limit 100 --json number,title,createdAt \
  | jq '.[] | select(.createdAt < "YYYY-MM-DD")'

# ラベルの頻度（繰り返し登場するテーマ＝センター候補）
gh issue list --limit 200 --json labels \
  | jq '[.[].labels[].name] | group_by(.) | map({label: .[0], count: length}) | sort_by(-.count)'
```

センター候補として読むもの：
- **繰り返し登場するテーマ**: 同じ概念が異なるIssueに何度も現れる
- **誰も着手しないIssue群**: 長期放置＝変容エネルギーが届いていないセンター
- **リオープンを繰り返すIssue**: 根が解決されていない潜在センターのシグナル
- **ラベルのクラスタ**: 同じラベルが集まる領域に共通の潜在センターがある

### Step 2: エージェントによるバックログセンターマップ（仮説）

```
## バックログセンターマップ（仮説）

繰り返し登場するテーマ（顕在センター候補）:
  - [テーマ名]: Issue #XX, #YY, #ZZ に共通して現れる
  - [テーマ名]: ラベル [label] が [N] 件に付いている

放置されているIssue群（弱いセンター候補）:
  - [Issue番号・タイトル]: [期間] 放置。関連する概念: [概念名]

共通の根を持つIssueクラスタ:
  - [クラスタ名]: Issue #XX, #YY は同じ問題を別の角度から指している可能性

潜在センター仮説:
  - [センター候補名]: [なぜ潜在していると読んだか]
```

### Step 3: 人間への問い返し — モヤモヤ感覚を聞く

エージェントの構造分析には「数字に現れない感覚」が抜け落ちる。
人間のモヤモヤを聞いて補正する。

> 「バックログを見ていて、感覚として聞かせてください。
>
> **① 見るたびに「これ、なんか気になる」と感じるIssueやテーマはありますか？**
>    うまく言えなくてもいいです。
>
> **② 「ずっと先送りにしてしまっている」ものはありますか？**
>    やりたくない・怖い・どこから手をつければいいかわからない——どれでも。
>
> **③ 「本当はここが一番大事なのに、バックログに入っていない」と感じることはありますか？**
>    言語化されていない、でも気になっている何か。
>
> 言葉にならなくてもかまいません。『あのあたり』でも受け取ります。」

```
モヤモヤ感覚の受け取り方:
①「気になる」     → 潜在センターの手がかり。バックログの外にあるかもしれない
②「先送り」       → 弱いセンター。恐れ起因か愛着起因かを確認する
③「入っていない」  → 隠れセンター。言語化を助ける問いを続ける
```

## Layer U: インターフェース×人のセンター検出プロトコル

Layer U のセンターはAIが単独で検出できる部分と、人間の感覚なしには検出できない部分がある。

### AIが分析できる範囲（構造的分析）

```bash
# UIコンポーネントの参照集中点（どのコンポーネントが多くの場所から使われているか）
grep -r "import\|from" . --include="*.tsx" --include="*.jsx" \
  | grep -v "node_modules\|test" \
  | sed "s/.*from '//; s/'.*//" | sort | uniq -c | sort -rn | head -20

# ページ・画面ファイルの一覧（フロー構造の把握）
find . -name "*.page.*" -o -name "*.screen.*" -o -name "*.view.*" \
  | grep -v "node_modules"
```

エージェントが構造から読めるUIセンター候補：
- 多くの画面から参照されるコンポーネント（構造的センター）
- ルーティングの分岐点（フローセンター）
- エラー表示・ローディング状態の集中点（弱さのセンター候補）
- ラベル・テキストの重複（共鳴の欠如）

### AIが単独では検出できないもの

- 実際の操作感・テンポ・引っかかり感
- 「なんか気持ち悪い」という直感
- 認知負荷の体感
- 「迷う」「戻りたくなる」という体験

**→ 人間の感覚が Layer U のセンター検出の主たる入力になる**

### 人間への問い返し — 操作感覚を聞く

STEP 3 の感覚マッピング（①〜④）を Layer U に特化して行う。

> 「このUI・画面・操作について、感覚として聞かせてください。
>
> **① 使うたびに「なんか引っかかる」「ここで止まる」場所はありますか？**
>
> **② 「壊れそう」「触りたくない」と感じる画面・操作はありますか？**
>    怖い・慣れるまで時間がかかった——恐れに近い感覚。
>
> **③ 「このUIのここが好き」「ここは残したい」と感じる部分はありますか？**
>    愛着がある、これがあるから使い続けている気がする。
>
> **④ 「こうなったら気持ちよく使える気がする」という感覚はありますか？**
>    まだそうなっていないけど、変わったらしっくりくる気がする。」

```
## UIセンターマップ（仮説 + 感覚入力）

エージェントの構造分析（仮説）:
  構造的センター: [多く参照されるコンポーネント]
  フローセンター: [ルーティング分岐点]
  弱さのセンター候補: [エラー・ローディング集中点]

人間の感覚入力:
  ①引っかかる（変容候補）: [場所・操作]
  ②壊れそう（脆いセンター）: [場所・操作]
  ③大事にしたい（生きているセンター）: [場所・操作]
  ④育てたい（潜在センター）: [場所・操作]

→ 感覚入力で仮説を補正した最終センターマップ:
  [補正後の内容]
```

## Layer H: 人間×エージェントのセンター検出プロトコル

Layer H のセンターは、会話そのものを分析対象にする。
人間が無意識に行っている発話パターンから、エージェントが読み取る。

### 会話から読めるシグナル

| 発話パターン | 読み取り |
|---|---|
| 同じ概念を言い直す・修正する | その概念がまだ共有されていないセンター |
| 「それじゃない」「違う」と返す | エージェントの認識とのズレのセンター |
| 話題が自然に速くなる・広がる | 共鳴しているセンター（生きている） |
| 言いかけて止まる・「まあいいか」 | 言語化されていない潜在センター |
| 同じ質問・懸念を繰り返す | エージェントが応えられていないセンター |
| 曖昧な言葉を使い続ける（「あのへん」） | まだ輪郭がないセンター（命名が助けになる） |

### エージェントの自己開示プロトコル

会話の節目（STEP 2 終了後 / STEP 5 振り返り時）に、エージェントが観察を開示する。

> 「この会話を通じて気づいたことを共有します。
>
> **何度も確認していた概念**: [●]
>    → ここがまだ共有できていないセンターかもしれません。
>
> **話が自然に広がった場所**: [▲]
>    → ここに生きているセンターがある気がします。
>
> **私が読み違えた箇所**: [■]（あれば）
>    → ここに、私がまだ見えていない構造があります。
>
> この観察はあなたの感覚と合っていますか？」

```
## H層センターマップ

共有できていない概念（ズレのセンター）:
  - [概念名]: [何度やり取りされたか / どんなズレがあったか]

共鳴したトピック（生きているセンター）:
  - [トピック]: [どのタイミングで話が広がったか]

言語化されていない潜在センター:
  - [仮の名前 or「名前なし」]: [人間がどんな言葉を使っていたか]

エージェントが応えられなかった問い:
  - [問いの内容]: [なぜ応えられなかったか]
```

## 4レイヤー統合センターマップ

各レイヤーのセンターを重ねて、共通の根を探す。

```
## センターマップ（4レイヤー統合）

対象: [システム / 機能 / 変容テーマ]

Layer C（コード）:
  顕在センター: [名前・強さ]
  潜在センター: [名前・理由]
  弱いセンター: [名前・原因]

Layer U（インターフェース）:
  構造的センター（AI分析）: [名前]
  生きているセンター（感覚③）: [場所・操作]
  弱いセンター（感覚①②）: [場所・操作]
  潜在センター（感覚④）: [場所・操作]

Layer P（プロダクト）:
  繰り返し登場するテーマ: [テーマ名]
  放置されているIssue群: [Issue番号]
  モヤモヤセンター（感覚入力）: [内容]
  潜在センター: [名前・理由]

Layer H（会話）:
  共有できていない概念: [概念名]
  共鳴したトピック: [トピック]
  言語化されていない潜在センター: [仮名]

複数レイヤーに共通して現れるセンター（最優先候補）:
  - [センター名]: C/U/P/H のどこに現れているか
```

複数レイヤーに同じセンターが現れるとき、それは全体性の核心に近い。

## DDD（ドメイン駆動設計）との対応

| DDDの概念 | センター分析での対応 |
|-----------|-------------------|
| 集約（Aggregate） | 力強いセンター（特性2）の実現形 |
| 境界づけられたコンテキスト | 境界（特性3）の大規模実現 |
| ユビキタス言語 | 共鳴（特性12）の言語的実現 |
| ドメインイベント | 深い相互結合（特性8）の実現手段 |
| 値オブジェクト | よい形（特性6）の具体的実装 |
